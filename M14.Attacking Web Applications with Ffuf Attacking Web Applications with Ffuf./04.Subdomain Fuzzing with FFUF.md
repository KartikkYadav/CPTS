# Subdomain Fuzzing with FFUF

## Overview

Subdomain fuzzing is the process of discovering hidden or unknown subdomains belonging to a target domain.

A subdomain is simply:
```text
A website that exists under another main domain
```

For example:

```text
photos.google.com
mail.google.com
admin.example.com
api.website.com
```

Here:
- `google.com` is the main domain
- `photos` and `mail` are subdomains

Subdomains are extremely important during reconnaissance because organizations often host:
- Admin portals
- APIs
- Development servers
- Testing environments
- Backup applications
- Internal dashboards

on separate subdomains.

---

# Why Subdomain Enumeration is Important

Many organizations secure their main website well, but forget about:
- Old subdomains
- Development systems
- Staging environments
- Internal tools

A forgotten subdomain may expose:
- Login panels
- Vulnerable applications
- Sensitive APIs
- Misconfigured services

This is why subdomain enumeration is a major part of:
- Bug bounty hunting
- External pentesting
- Red teaming
- CPTS exams

---

# How Subdomain Fuzzing Works

Subdomain fuzzing works by testing:
```text
Different possible subdomain names
```

against a target domain.

FFUF sends requests such as:

```text
admin.inlanefreight.com
blog.inlanefreight.com
mail.inlanefreight.com
api.inlanefreight.com
```

and checks whether:
- DNS records exist
- The server responds
- A website is hosted there

---

# Required Components

To perform subdomain fuzzing, we need:
1. A target domain
2. A subdomain wordlist

---

# Subdomain Wordlists

SecLists contains DNS-specific wordlists located in:

```text
/opt/useful/seclists/Discovery/DNS/
```

Common wordlists include:
- `subdomains-top1million-5000.txt`
- `subdomains-top1million-20000.txt`

Smaller wordlists:
- Faster
- Less noisy

Larger wordlists:
- Better coverage
- More requests
- Longer scans

---

# Basic Subdomain Fuzzing Command

```bash
ffuf -w /opt/useful/seclists/Discovery/DNS/subdomains-top1million-5000.txt:FUZZ -u https://FUZZ.inlanefreight.com/
```

---

# Understanding the Command

This command tells FFUF to:
- Use the DNS wordlist
- Replace `FUZZ`
- Test each possible subdomain

Examples generated:

```text
https://admin.inlanefreight.com
https://blog.inlanefreight.com
https://support.inlanefreight.com
```

---

# Example Results

```text
FUZZ: support
FUZZ: ns3
FUZZ: blog
FUZZ: www
```

---

# Understanding the Results

A response such as:

```text
Status: 200
```

means:
```text
The subdomain exists and is accessible
```

A response such as:

```text
301 Redirect
```

usually means:
```text
The subdomain exists and redirects elsewhere
```

---

# Important Observation

Even a:
```text
403 Forbidden
```

response is valuable because it means:
```text
The subdomain exists
```

but access is restricted.

---

# Why Some Domains Return No Results

When scanning:

```text
academy.htb
```

FFUF returned:
```text
Errors: 4997
```

This does NOT automatically mean:
```text
No subdomains exist
```

---

# Public DNS vs Local Hosts File

This happens because:
```text
academy.htb is not publicly resolvable
```

Even if:
```text
academy.htb
```

exists in:
```text
/etc/hosts
```

only the main domain was added.

FFUF attempts to resolve:

```text
admin.academy.htb
dev.academy.htb
test.academy.htb
```

through:
```text
Public DNS
```

Since public DNS has no records for them:
- Resolution fails
- FFUF returns errors

---

# Important Concept

FFUF subdomain fuzzing relies on:
```text
DNS resolution
```

If the target is:
- Internal
- Lab-based
- HTB-only
- Not publicly hosted

then public DNS will not know about the subdomains.

# Common Real-World Subdomains

Subdomain fuzzing commonly reveals:
- `admin`
- `dev`
- `staging`
- `api`
- `mail`
- `vpn`
- `test`
- `beta`

These are often high-value targets.

---

# Common HTTP Response Codes

| Status | Meaning |
|------|----------|
| 200 | Accessible |
| 301/302 | Redirect |
| 401 | Authentication required |
| 403 | Exists but forbidden |
| 404 | Not found |

---

# Important FFUF Options

| Option | Purpose |
|--------|----------|
| `-w` | Wordlist |
| `-u` | Target URL |
| `-mc` | Match status codes |
| `-fc` | Filter status codes |
| `-t` | Threads |
| `-v` | Verbose output |

---

# Performance Considerations

Subdomain fuzzing can generate:
- Thousands of DNS requests
- Large numbers of HTTP requests

Larger wordlists increase:
- Discovery chances
- Scan duration
- Noise

---

# Key Points

Subdomain fuzzing is used to discover hidden subdomains by testing common names against a target domain and checking whether valid DNS records and HTTP responses exist. It is one of the most important reconnaissance techniques because organizations frequently expose development systems, APIs, and admin portals through forgotten or hidden subdomains.
