# VHost Fuzzing with FFUF

## Overview

During normal subdomain fuzzing, FFUF discovers:
```text
Publicly resolvable subdomains
```

This works because public DNS records exist for domains like:

```text
blog.example.com
mail.example.com
api.example.com
```

However, many organizations also use:
```text
Internal or non-public virtual hosts (VHosts)
```

These do NOT always exist in public DNS records.

Because of this, traditional subdomain fuzzing may completely miss them.

To discover these hidden virtual hosts, we use:
```text
VHost Fuzzing
```

---

# What is a VHost?

A VHost (Virtual Host) is:
```text
A separate website hosted on the same server/IP address
```

This means:
- One server
- One IP
- Multiple websites

can all exist together.

---

# Example

A single server IP may host:

```text
www.company.com
admin.company.com
dev.company.com
api.company.com
```

All on:
```text
The same IP address
```

The web server decides which website to display based on:
```text
Host header
```

inside the HTTP request.

---

# VHosts vs Subdomains

## Subdomains

Subdomains usually require:
```text
Public DNS records
```

Examples:

```text
blog.google.com
mail.google.com
```

DNS translates these names into IP addresses.

---

## VHosts

VHosts may:
- Exist publicly
- Exist internally only
- Not have public DNS records

Even without public DNS:
```text
The web server may still respond to them
```

if the correct:
```text
Host header
```

is supplied.

---

# Why VHost Fuzzing Matters

Organizations often hide:
- Admin portals
- Development environments
- Staging servers
- Internal APIs

behind:
```text
Non-public VHosts
```

These may not appear in:
- DNS enumeration
- Public recon
- Search engines

but still exist on the target server.

---

# Why Normal Subdomain Fuzzing Fails

Suppose we fuzz:

```text
admin.academy.htb
```

FFUF first attempts:
```text
DNS resolution
```

If no public DNS record exists:
- Resolution fails
- Request never reaches server

So the hidden VHost remains undiscovered.

---

# How VHost Fuzzing Works

Instead of fuzzing DNS records, we:
```text
Directly fuzz the Host header
```

This allows us to:
- Send requests directly to the target IP
- Pretend we are requesting different hostnames

The server then decides:
```text
Which virtual host to serve
```

---

# Important HTTP Header

VHost fuzzing relies on:

```http
Host:
```

header.

Example:

```http
Host: admin.academy.htb
```

---

# VHost Fuzzing Command

```bash
ffuf -w /opt/useful/seclists/Discovery/DNS/subdomains-top1million-5000.txt:FUZZ -u http://academy.htb:PORT/ -H 'Host: FUZZ.academy.htb'
```

---

# Understanding the Command

This command tells FFUF to:
1. Use the subdomain wordlist
2. Send requests to:
   ```text
   academy.htb
   ```
3. Modify the:
   ```http
   Host:
   ```
   header for every request

Examples generated:

```http
Host: admin.academy.htb
Host: dev.academy.htb
Host: mail.academy.htb
```

---

# Important Difference from Subdomain Fuzzing

During normal subdomain fuzzing:
```text
DNS resolution is required
```

During VHost fuzzing:
```text
Only the target IP/server is needed
```

This allows discovery of:
- Hidden VHosts
- Internal applications
- Non-public services

---

# Example Results

```text
mail2
dns2
webmail
static
web
```

All returned:
```text
Status: 200
```

---

# Important Problem

At first glance, it appears:
```text
Every VHost exists
```

But this is misleading.

---

# Why Every Response Returns 200

We are always visiting:

```text
http://academy.htb:PORT/
```

The only thing changing is:
```http
Host:
```

header.

The server may simply:
- Return the default webpage
- Ignore invalid hostnames

This causes:
```text
False positives
```

---

# Detecting Real VHosts

The key indicator is usually:
```text
Different response size
```

If the VHost is real:
- The server serves different content
- Response size changes
- Word count changes
- Line count changes

---

# Example

Most responses:

```text
Size: 900
```

But one response may return:

```text
Size: 1450
```

This strongly suggests:
```text
A real VHost exists
```

---

# Important FFUF Response Fields

| Field | Meaning |
|------|----------|
| Status | HTTP status code |
| Size | Response size |
| Words | Word count |
| Lines | Line count |

These help identify:
- Real VHosts
- Default responses
- False positives

---

# Filtering False Positives

Often we filter:
```text
Default response sizes
```

using:

```bash
-fs
```

---

# Example

```bash
-fs 900
```

This removes:
```text
Responses with size 900
```

making real VHosts easier to spot.

---

# Useful FFUF Options

| Option | Purpose |
|--------|----------|
| `-w` | Wordlist |
| `-u` | Target URL |
| `-H` | Add HTTP header |
| `-fs` | Filter response size |
| `-mc` | Match status codes |
| `-v` | Verbose output |

---

# Common Hidden VHosts

VHost fuzzing commonly discovers:
- `admin`
- `dev`
- `staging`
- `test`
- `beta`
- `api`
- `internal`

These are often high-value targets.

---

# Real-World Importance

VHost fuzzing is extremely useful during:
- External pentests
- Bug bounty recon
- Internal assessments
- HTB labs
- CPTS exams

because many hidden applications exist only as:
```text
Internal virtual hosts
```

and are invisible to normal DNS enumeration.

---

# Key Points

VHost fuzzing discovers hidden websites hosted on the same server by fuzzing the HTTP Host header instead of relying on public DNS records. This technique is important because many internal or hidden applications do not have public DNS entries but still respond when the correct Host header is supplied.

---

# One-Line Revision

VHost fuzzing with FFUF discovers hidden virtual hosts by fuzzing the HTTP Host header against a known target IP or domain without relying on public DNS records.
