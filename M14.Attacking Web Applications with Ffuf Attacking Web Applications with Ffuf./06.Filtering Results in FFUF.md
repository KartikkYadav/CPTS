# Filtering Results in FFUF

## Overview

During fuzzing, FFUF can generate:
- Hundreds
- Thousands
- Sometimes millions

of responses.

## Additional Autimatic tool:

    gobuster vhost -u http://academy.htb:30763 -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt  --append-domain  

Not every response is useful.

Many web servers return:
```text
200 OK
```

even for:
- Invalid pages
- Invalid VHosts
- Non-existing paths

This creates:
```text
False positives
```

which can make analysis difficult.

To solve this problem, FFUF provides:
```text
Filtering and Matching options
```

that help isolate meaningful results.

---

# Why Filtering is Important

Suppose we are performing:
```text
VHost fuzzing
```

and every request returns:
```text
200 OK
```

At first glance, it may appear:
```text
Every VHost exists
```

But in reality:
- The server may simply return the default page
- Invalid requests may still receive a successful response

This makes raw results unreliable.

Filtering allows us to:
- Remove noisy responses
- Eliminate false positives
- Focus only on unique results

---

# Matcher vs Filter

FFUF supports two concepts:
- Matchers
- Filters

---

# Matchers

Matchers tell FFUF:
```text
Which responses to KEEP
```

---

# Filters

Filters tell FFUF:
```text
Which responses to REMOVE
```

---

# Important FFUF Matching Options

| Option | Purpose |
|------|----------|
| `-mc` | Match HTTP status codes |
| `-ml` | Match line count |
| `-ms` | Match response size |
| `-mw` | Match word count |
| `-mr` | Match regex |

---

# Important FFUF Filtering Options

| Option | Purpose |
|------|----------|
| `-fc` | Filter status codes |
| `-fl` | Filter line count |
| `-fs` | Filter response size |
| `-fw` | Filter word count |
| `-fr` | Filter regex |

# Common Problem During VHost Fuzzing

When fuzzing VHosts:

```bash
ffuf -w wordlist.txt:FUZZ -u http://academy.htb:PORT/ -H 'Host: FUZZ.academy.htb'
```

every request may return:

```text
Status: 200
Size: 900
```

This happens because:
```text
The server returns the default page for invalid VHosts
```

As a result:
- All responses look valid
- The output becomes noisy

---

# Identifying False Positives

The important observation is:
```text
All invalid responses have the same response size
```

For example:

```text
Size: 900
```

This means:
```text
900-byte responses are probably default pages
```

---

# Using Response Size Filtering

We can remove these noisy responses using:

```bash
-fs 900
```

This tells FFUF:
```text
Filter out every response with size 900
```

---

# Filtered VHost Fuzzing Command

```bash
ffuf -w /opt/useful/seclists/Discovery/DNS/subdomains-top1million-5000.txt:FUZZ -u http://academy.htb:PORT/ -H 'Host: FUZZ.academy.htb' -fs 900
```

---

# Understanding the Command

This command:
1. Performs VHost fuzzing
2. Sends different Host headers
3. Removes responses with size 900

This helps expose:
```text
Responses that are different from the default page
```

---

# Example Result

```text
admin [Status: 200, Size: 0]
```

This is important because:
```text
The response size is different
```

which strongly suggests:
```text
A real VHost exists
```

---

# Why Response Size Matters

Even if:
```text
HTTP status code is the same
```

the:
- Size
- Words
- Lines

may differ.

This difference often reveals:
- Real pages
- Hidden VHosts
- Valid endpoints

---

# Verifying the VHost

After discovering:

```text
admin.academy.htb
```

we add it to:
```text
/etc/hosts
```

Example:

```text
SERVER_IP admin.academy.htb
```

Then visit:

```text
https://admin.academy.htb:PORT/
```

---

# Important Observation

The page may appear:
- Blank
- Empty
- Different from main website

This confirms:
```text
We reached a different VHost
```

---

# Additional Verification

Visiting:

```text
https://admin.academy.htb:PORT/blog/index.php
```

returns:
```text
404 NOT FOUND
```

while the original VHost may contain that page.

This further confirms:
```text
The VHost serves different content
```

# Common Filtering Methods

## Filter Status Codes

```bash
-fc 404
```

Removes:
```text
404 Not Found
```

responses.

---

# Filter Response Size

```bash
-fs 900
```

Removes responses matching:
```text
900 bytes
```

---

# Filter Word Count

```bash
-fw 423
```

Filters responses with:
```text
423 words
```

---

# Filter Line Count

```bash
-fl 56
```

Filters responses with:
```text
56 lines
```

---

# Matching Instead of Filtering

Sometimes instead of removing responses, we want to:
```text
Only show specific results
```

Examples:

## Match Status Codes

```bash
-mc 200
```

---

# Match Response Size

```bash
-ms 1500
```

---

# Why Filtering is Critical in Real Engagements

Real-world applications often:
- Return fake 200 responses
- Redirect all invalid requests
- Use wildcard routing

Without filtering:
- Results become unusable
- False positives increase heavily
- Important findings may be missed

Filtering improves:
- Accuracy
- Visibility
- Recon efficiency

---

# Key Points

Filtering in FFUF is essential for removing noisy or default responses during fuzzing. Instead of relying only on HTTP status codes, analyzing response size, word count, and line count helps identify real pages, directories, and VHosts while eliminating false positives.

---

# One-Line Revision

FFUF filtering removes noisy or duplicate responses using response size, word count, line count, and status codes to identify real findings more accurately during fuzzing.
