# Parameter Fuzzing - POST Requests & Value Fuzzing

# Overview

After discovering hidden:
- Directories
- Pages
- VHosts
- GET parameters

the next important step in web fuzzing is:
```text
POST parameter fuzzing
```

Modern web applications heavily rely on POST requests for:
- Login forms
- Admin panels
- APIs
- Search functionality
- File uploads
- User actions

Unlike GET requests, POST parameters are not visible directly in the URL. They are hidden inside:
```text
The HTTP request body
```

Because of this, hidden POST parameters are often:
- Forgotten by developers
- Poorly secured
- Less tested
- Not publicly documented

This makes POST parameter fuzzing extremely useful during:
- Web pentesting
- Bug bounty hunting
- API testing
- CPTS exams

---

# Understanding GET vs POST Parameters

## GET Parameters

GET parameters are passed directly in the URL.

Example:

```text
/admin.php?id=1
```

Here:
- `id` is the parameter name
- `1` is the value

GET requests are easy to notice because the parameters are visible.

---

## POST Parameters

POST parameters are sent inside:
```text
The HTTP request body
```

and do not appear in the URL.

Example:

```http
POST /admin.php

id=1
```

POST requests are commonly used because they:
- Hide parameters from the URL
- Support larger data
- Are used in forms and APIs

---

# Why POST Parameter Fuzzing is Important

Applications often contain:
- Hidden POST parameters
- Internal API fields
- Deprecated functionality
- Developer-only features
- Debug variables

These parameters may not appear:
- In HTML forms
- In JavaScript
- In page source
- Anywhere visible publicly

But the backend may still process them if manually supplied.

This is why parameter fuzzing is powerful.

Suppose we discovered:

```text
/admin/admin.php
```

When we visit the page:
```text
Access is denied
```

This suggests:
```text
The application expects additional input
```

Possibilities include:
- Authentication
- Session cookies
- GET parameters
- POST parameters

After GET fuzzing, we now attempt:
```text
POST parameter fuzzing
```

# POST Parameter Fuzzing with FFUF

To fuzz POST parameters, FFUF needs:
1. POST request method
2. Request body data
3. Correct Content-Type header

---

# Using POST Requests in FFUF

POST requests are enabled using:

```bash
-X POST
```

This tells FFUF:
```text
Send HTTP POST requests instead of GET
```

---

# Sending POST Data

POST data is supplied using:

```bash
-d
```

Example:

```bash
-d 'id=1'
```

---

# Important PHP Content-Type Header

PHP applications usually expect POST data in:

```http
application/x-www-form-urlencoded
```

format.

Without the correct header:
- Parameters may not parse properly
- The application may ignore the request

---

# Adding Content-Type Header

We specify it using:

```bash
-H 'Content-Type: application/x-www-form-urlencoded'
```

---

# POST Parameter Fuzzing Command

```bash
ffuf -w /opt/useful/seclists/Discovery/Web-Content/burp-parameter-names.txt:FUZZ -u http://admin.academy.htb:PORT/admin/admin.php -X POST -d 'FUZZ=key' -H 'Content-Type: application/x-www-form-urlencoded' -fs xxx
```

---

# Understanding the Command

This command tells FFUF to:
1. Use the Burp parameter wordlist
2. Replace `FUZZ` with parameter names
3. Send POST requests like:

```text
admin=key
id=key
debug=key
token=key
```

4. Filter default responses using:
```bash
-fs
```

---

# Why Filtering is Important

Most parameter attempts return:
```text
The exact same response
```

For example:
- Same access denied page
- Same error page
- Same response size

Without filtering:
```text
Thousands of false positives appear
```

Filtering removes:
```text
Default responses
```

so we can focus only on:
```text
Unique or interesting responses
```

---

# Example Result

```text
id [Status: 200, Size: xxx]
```

This is important because:
```text
The application responded differently to the id parameter
```

which strongly suggests:
```text
id is a real parameter
```

---

# Verifying the Parameter Manually

After discovering:
```text
id
```

we manually test it using:
```bash
curl
```

---

# CURL Verification Command

```bash
curl http://admin.academy.htb:PORT/admin/admin.php -X POST -d 'id=key' -H 'Content-Type: application/x-www-form-urlencoded'
```

---

# Example Response

```text
Invalid id!
```

This response is extremely important.

Why?

Because:
```text
The application processed the parameter
```

instead of ignoring it.

This confirms:
```text
id is a valid backend parameter
```

# What Comes Next?

After finding a valid parameter, the next step is:
```text
Value fuzzing
```

Now we no longer fuzz:
- Parameter names

Instead we fuzz:
- Parameter values

---

# What is Value Fuzzing?

Value fuzzing means:
```text
Testing many possible values for a valid parameter
```

Examples:

```text
id=1
id=2
id=3
```

The goal is to discover:
- Valid objects
- Hidden resources
- Sensitive data
- Admin functionality

---

# Why Value Fuzzing is Powerful

Applications often use:
- Sequential IDs
- Predictable tokens
- Numeric references
- Internal object identifiers

If we guess the correct value, we may access:
- Other users' data
- Hidden records
- Sensitive content
- Flags in labs

This technique is heavily related to:
```text
IDOR vulnerabilities
```

---

# Building a Custom Wordlist

For parameter values, pre-made wordlists may not always work.

We often create:
```text
Custom wordlists
```

based on:
- Expected input type
- Application logic
- Numeric patterns

---

# Example Scenario

The parameter:
```text
id
```

likely expects:
```text
Numbers
```

Possible values:

```text
1
2
3
100
999
```

---

# Creating a Numeric Wordlist

The easiest method is using Bash:

```bash
for i in $(seq 1 1000); do echo $i >> ids.txt; done
```

---

# Understanding the Command

This command:
1. Generates numbers from 1 to 1000
2. Writes them into:
```text
ids.txt
```

---

# Example Wordlist Content

```text
1
2
3
4
5
```

---

# Value Fuzzing Command

```bash
ffuf -w ids.txt:FUZZ -u http://admin.academy.htb:PORT/admin/admin.php -X POST -d 'id=FUZZ' -H 'Content-Type: application/x-www-form-urlencoded' -fs xxx
```

---

# Understanding the Command

This command:
1. Uses numbers from:
```text
ids.txt
```

2. Replaces:
```text
FUZZ
```

3. Sends requests like:

```text
id=1
id=2
id=3
```

4. Filters default responses

---

# Identifying Valid Values

If one value produces:
- Different response size
- Different word count
- Different content

then it likely indicates:
```text
A valid ID or hidden object
```

---

# Example Discovery

A valid response may:
- Reveal hidden content
- Return admin data
- Expose flags
- Display sensitive information

This confirms:
```text
The correct value was discovered
```

---

# Manual Verification

After finding the valid value, we manually verify it using:
```bash
curl
```

Example:

```bash
curl http://admin.academy.htb:PORT/admin/admin.php -X POST -d 'id=VALID_ID' -H 'Content-Type: application/x-www-form-urlencoded'
```

# Real-World Importance

POST parameter and value fuzzing are extremely important because modern applications heavily rely on hidden backend parameters and object identifiers.

This technique is commonly used for:
- API testing
- IDOR testing
- Admin panel discovery
- Hidden feature discovery
- Access control testing

# Key Points

POST parameter fuzzing discovers hidden backend parameters by fuzzing POST request bodies, while value fuzzing identifies valid values accepted by those parameters. Together, these techniques help uncover hidden functionality, sensitive resources, and vulnerable application logic that may not be visible through normal browsing.

---

# One-Line Revision

POST parameter fuzzing discovers hidden POST parameters, and value fuzzing identifies accepted parameter values that may expose hidden application functionality or sensitive data.
