# Parameter Fuzzing - GET Requests with FFUF

## Overview

After discovering hidden directories and VHosts, the next step during web reconnaissance is often:
```text
Parameter fuzzing
```

Many web applications use:
- GET parameters
- POST parameters

to:
- Control functionality
- Authenticate users
- Access hidden features
- Load internal resources

Sometimes developers leave behind:
- Hidden parameters
- Deprecated functionality
- Debug features
- Unpublished endpoints

These parameters may not appear:
- In the webpage
- In forms
- In JavaScript
- In source code

but they may still work if manually supplied.

Parameter fuzzing helps discover these hidden inputs.

---

# Why Parameter Fuzzing is Important

Hidden parameters are extremely valuable because they may expose:
- Admin functionality
- Sensitive data
- Debug modes
- Access control bypasses
- Vulnerable features

These parameters are often:
```text
Poorly tested and poorly secured
```

because developers assume:
```text
Nobody knows they exist
```

This makes parameter fuzzing a powerful technique during:
- Web pentesting
- Bug bounty hunting
- CPTS exams
- API testing

---

# Scenario

Suppose we discovered:

```text
http://admin.academy.htb:PORT/admin/admin.php
```

When visiting the page, we receive:

```text
You don't have access to read the flag
```

This indicates:
```text
The page expects additional input
```

Possibilities include:
- Authentication cookies
- Session tokens
- GET parameters
- POST parameters

Since:
- We are not logged in
- No authentication exists

there is a strong possibility that:
```text
A hidden parameter controls access
```

---

# Understanding GET Parameters

GET parameters are passed:
```text
Inside the URL
```

using:
```text
?
```

symbol.

Example:

```text
/admin.php?user=admin
```

Here:
- `user` is the parameter name
- `admin` is the value

Applications commonly use GET parameters for:
- Search queries
- User IDs
- Debug modes
- Feature toggles
- Access keys

---

# Parameter Fuzzing Concept

Instead of fuzzing:
- Directories
- Filenames
- VHosts

we now fuzz:
```text
Parameter names
```

FFUF replaces:
```text
FUZZ
```

with possible parameter names from a wordlist.

---

# Parameter Wordlist

SecLists provides a parameter wordlist:

```text
/opt/useful/seclists/Discovery/Web-Content/burp-parameter-names.txt
```

This contains:
- Common developer parameter names
- API parameters
- Debug variables
- Authentication-related names

---

# GET Parameter Fuzzing Command

```bash
ffuf -w /opt/useful/seclists/Discovery/Web-Content/burp-parameter-names.txt:FUZZ -u http://admin.academy.htb:PORT/admin/admin.php?FUZZ=key -fs xxx
```

---

# Understanding the Command

This command tells FFUF to:
1. Use the parameter wordlist
2. Replace `FUZZ`
3. Send requests like:

```text
/admin.php?debug=key
/admin.php?admin=key
/admin.php?access=key
```

4. Filter default responses using:
```bash
-fs
```

---

# Why Filtering is Needed

Most parameter attempts return:
```text
The same response
```

For example:
- Same error page
- Same access denied page
- Same response size

Without filtering:
```text
Thousands of useless results appear
```

Filtering removes:
```text
Default responses
```

so only unique responses remain.

---

# Example Filtering

```bash
-fs 798
```

This removes responses with:
```text
798-byte response size
```

which usually corresponds to:
```text
The default "access denied" page
```

---

# Identifying Valid Parameters

If one parameter returns:
- Different response size
- Different word count
- Different content

then it may indicate:
```text
A valid hidden parameter
```

---

# Example Result

```text
[Status: 200, Size: xxx]
```

A different size strongly suggests:
```text
The application processed the parameter
```

instead of ignoring it.

---

# Testing the Parameter Manually

After finding a valid parameter, we manually test it:

```text
/admin/admin.php?REDACTED=key
```

This helps verify:
- Whether the parameter works
- Whether functionality changes
- Whether sensitive content becomes accessible

---

# Example Observation

The page responds with:

```text
This method is deprecated
```

This indicates:
- The parameter was once valid
- The functionality may no longer be active
- Hidden developer functionality existed previously

Even deprecated parameters are important because they may still:
- Trigger hidden code paths
- Expose vulnerabilities
- Reveal old logic

---

# Why Hidden Parameters are Dangerous

Developers frequently:
- Forget to remove old functionality
- Leave hidden debug parameters
- Disable features incorrectly

Attackers can abuse these hidden parameters for:
- Authentication bypass
- Local File Inclusion (LFI)
- SQL Injection
- Remote Code Execution
- Information disclosure

---

# Important FFUF Options

| Option | Purpose |
|------|----------|
| `-w` | Wordlist |
| `-u` | Target URL |
| `-fs` | Filter response size |
| `-fc` | Filter status codes |
| `-fw` | Filter word count |
| `-mc` | Match status codes |

---

# Important Observation

A parameter does NOT need to:
- Appear in forms
- Exist in source code
- Be linked publicly

to still function.

Many hidden parameters remain:
```text
Accessible directly through manual requests
```

---

# Real-World Importance

Parameter fuzzing is extremely useful because modern applications often contain:
- Hidden APIs
- Internal debug parameters
- Legacy developer functionality
- Deprecated endpoints

that are not visible through normal browsing.

---

# Key Points

GET parameter fuzzing with FFUF discovers hidden URL parameters by replacing parameter names with common words from a wordlist and analyzing differences in responses. Hidden parameters are valuable because they often expose internal functionality, legacy code paths, or poorly secured developer features.

---

# One-Line Revision

GET parameter fuzzing with FFUF discovers hidden URL parameters by fuzzing parameter names and analyzing response differences to uncover hidden or vulnerable functionality.
