# Footprinting Lab - Medium

## Objective

The target server was accessible to everyone on the internal network.  
The goal was to enumerate the server, identify weaknesses, and obtain the credentials for the user `HTB`.

---

# Nmap Enumeration

Initial scan:

```bash
nmap -sC -sV 10.129.x.x
```

## Open Ports

| Port | Service | Purpose |
|---|---|---|
| 111 | rpcbind | NFS/RPC mapping |
| 135 | MSRPC | Windows RPC |
| 139 | NetBIOS | SMB legacy support |
| 445 | SMB | File sharing |
| 2049 | NFS | Network File System |
| 3389 | RDP | Remote Desktop |
| 5985 | WinRM | PowerShell remoting |

---

# SMB Enumeration

Initially, I attempted to identify SMB vulnerabilities such as:

- MS17-010 (EternalBlue)
- SMB misconfigurations
- Null sessions

However, the SMB service was not vulnerable.

Example check:

```bash
nmap --script smb-vuln-ms17-010 -p445 10.129.x.x
```

Result:

```text
NOT VULNERABLE
```

---

# RDP Enumeration

After SMB enumeration did not reveal an easy entry point, I moved to investigating RDP-related weaknesses and accessible services.

---

# NFS Enumeration

The NFS service turned out to be the most interesting attack surface.

## Enumerating NFS Exports

```bash
showmount -e 10.129.x.x
```

Result:

```text
/TechSupport (everyone)
```

This indicated that the NFS share was accessible anonymously.

---

# Mounting the NFS Share

Created a local mount point:

```bash
mkdir target-NFS
```

Mounted the share:

```bash
sudo mount -t nfs 10.129.x.x:/TechSupport ./target-NFS -o nolock
```

---

# Credential Discovery

Inside the mounted NFS share, I discovered support ticket files containing sensitive information.

One of the files exposed credentials:

```text
alex : lol123!mD
```

This demonstrated insecure credential handling and poor operational security.

---

# SMB Authentication

Using the discovered credentials, I authenticated successfully to SMB:

```bash
smbclient -L //10.129.x.x -U alex
```

Accessible shares included:

```text
ADMIN$
C$
devshare
Users
IPC$
```

---

# SMB Share Enumeration

After enumerating the SMB shares, I discovered files related to SQL Server administration.

One important file contained SQL Server credentials:

```text
sa:87N1ns@slls83
```

This exposed the SQL Server administrator account credentials.

---

# RDP Access

Using the credentials obtained from the NFS share, I authenticated successfully to RDP:

```bash
xfreerdp /u:alex /p:'lol123!mD' /v:10.129.x.x
```

---

# SQL Server Enumeration

Inside the RDP session, I found:

- Microsoft SQL Server Management Studio (SSMS)

I used the SQL credentials found through SMB enumeration:

| Username | Password |
|---|---|
| sa | 87N1ns@slls83 |

and authenticated to SQL Server using:

```text
Authentication: SQL Server Authentication
Login: sa
Password: 87N1ns@slls83
```

---

# Credential Extraction

After connecting to SQL Server, I enumerated databases and extracted the credentials for the `HTB` user.

---

# Security Issues Identified

| Vulnerability | Impact |
|---|---|
| Anonymous NFS access | Sensitive file disclosure |
| Support tickets storing plaintext credentials | Credential compromise |
| SMB accessible with exposed credentials | Internal lateral movement |
| SQL administrator credentials stored insecurely | Full database compromise |
| RDP accessible with reused credentials | Remote system access |
| Weak internal segmentation | Multiple service compromise |

---

# Attack Path Summary

1. Performed Nmap enumeration
2. Investigated SMB vulnerabilities
3. Enumerated NFS exports
4. Mounted anonymous NFS share
5. Discovered credentials in support ticket files
6. Authenticated to SMB using discovered credentials
7. Found SQL Server administrator credentials in SMB shares
8. Logged into RDP using exposed credentials
9. Opened SQL Server Management Studio
10. Authenticated as `sa`
11. Extracted credentials for the `HTB` user

---

# Key Takeaway

This lab demonstrated how exposed internal services and insecure credential storage can lead to complete compromise without exploiting software vulnerabilities.

The primary weaknesses were:

- Anonymous NFS exposure
- Plaintext credentials in support tickets
- Credential reuse across services
- Insecure SQL credential storage

Even without exploiting a traditional vulnerability, sensitive information disclosure allowed full access to the environment.
