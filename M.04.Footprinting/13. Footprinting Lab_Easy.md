# Footprinting Lab - Easy

## Objective

We were tasked with performing enumeration against an internal server belonging to Inlanefreight Ltd.

The goal was to:
- Gather as much information as possible
- Avoid aggressive exploitation
- Identify security weaknesses
- Retrieve the `flag.txt` file

The following credentials were provided:

```text
ceil:qwer1234
```

---

# Nmap Enumeration

Initial scan:

```bash
nmap -sC -sV 10.129.107.65
```

## Results

| Port | State | Service | Version |
|---|---|---|---|
| 21 | open | ftp | ProFTPD |
| 22 | open | ssh | OpenSSH 8.2p1 Ubuntu 4ubuntu0.2 |
| 53 | open | domain | ISC BIND 9.16.1 |
| 2121 | open | ftp | FTP Service |

---

# SSH Enumeration

Attempted SSH login using provided credentials:

```bash
ssh ceil@10.129.107.65
```

Result:

```text
Permission denied (publickey)
```

## Observation

This indicated:
- User `ceil` exists
- Password authentication is disabled
- SSH key authentication is required

---

# FTP Enumeration

## FTP on Port 21

Connected to the default FTP service:

```bash
ftp 10.129.107.65
```

After enumeration, the FTP directory appeared mostly empty.

---

## FTP on Port 2121

Connected to the alternate FTP service:

```bash
ftp 10.129.107.65 2121
```

Logged in using:

```text
Username: ceil
Password: qwer1234
```

After enumerating directories:

```bash
dir -a
```

I discovered:

```text
.ssh
```

---

# SSH Key Discovery

Navigated into the `.ssh` directory:

```bash
cd .ssh
dir -a
```

Discovered a private SSH key:

```text
id_rsa
```

Downloaded the key:

```bash
get id_rsa
```

---

# SSH Access Using Private Key

Changed key permissions locally:

```bash
chmod 600 id_rsa
```

Authenticated successfully using the key:

```bash
ssh -i id_rsa ceil@10.129.107.65
```

This provided shell access as user `ceil`.

---

# Flag Retrieval

After gaining shell access, I located and read the flag:

```bash
find / -name flag.txt 2>/dev/null
cat flag.txt
```

Successfully retrieved the flag.

---

# Security Issues Identified

| Vulnerability | Impact |
|---|---|
| FTP exposing user directories | Information disclosure |
| SSH private key exposed via FTP | Full account compromise |
| Weak service segregation | Increased attack surface |
| Alternate FTP service on port 2121 | Hidden attack vector |
| Improper sensitive file storage | Credential compromise |

---

# Key Takeaway

The primary issue was not SSH itself.

The critical vulnerability was:
- exposure of private SSH keys through FTP access

Even though password authentication was disabled for SSH, access was still possible because the private key was improperly accessible.

This demonstrates the importance of:
- proper file permissions
- secure storage of SSH keys
- restricting FTP access
- minimizing exposed services

---

# Attack Path Summary

1. Enumerated open ports with Nmap
2. Identified alternate FTP service on port 2121
3. Logged into FTP using provided credentials
4. Enumerated directories and discovered `.ssh`
5. Downloaded exposed private SSH key
6. Used key to authenticate to SSH
7. Retrieved `flag.txt`
