# Mitigating SQL Injection

We have learned how SQL Injection vulnerabilities occur and how attackers exploit them. However, understanding exploitation is only half of the picture. As security professionals, it is equally important to understand how SQL Injection can be prevented and how vulnerable applications can be secured.

There is no single defense against SQL Injection. Instead, secure applications rely on multiple layers of protection such as input sanitization, input validation, least-privilege database accounts, Web Application Firewalls (WAFs), and parameterized queries.

---

# Input Sanitization

One of the most common causes of SQL Injection is directly inserting user-supplied data into SQL queries.

Consider the following vulnerable PHP code:

```php
$username = $_POST['username'];
$password = $_POST['password'];

$query = "SELECT * FROM logins WHERE username='". $username ."' AND password='". $password ."';";
```

The application takes values from the POST request and inserts them directly into the SQL query. Since no filtering or escaping occurs, an attacker can inject malicious SQL syntax and alter the query logic.

To mitigate this issue, user input should be sanitized before being used in a query.

Example:

```php
$username = mysqli_real_escape_string($conn, $_POST['username']);
$password = mysqli_real_escape_string($conn, $_POST['password']);

$query = "SELECT * FROM logins WHERE username='". $username ."' AND password='". $password ."';";
```

The function `mysqli_real_escape_string()` escapes special characters such as:

```text
'
"
\
NULL
```

As a result, user input is treated as data rather than executable SQL syntax.

For PostgreSQL, a similar function is:

```php
pg_escape_string()
```

Input sanitization helps prevent many basic SQL Injection attacks by ensuring that special characters lose their SQL meaning.

# Input Validation

Input validation ensures that user input matches the format expected by the application before it reaches the database.

For example, consider the following vulnerable code:

```php
$q = "SELECT * FROM ports WHERE port_code ILIKE '%" . $_GET["port_code"] . "%'";
```

The application expects a port code consisting only of letters and spaces. Since no validation is performed, attackers can inject SQL statements.

A safer implementation uses regular expressions:

```php
$pattern = "/^[A-Za-z\s]+$/";
$code = $_GET["port_code"];

if(!preg_match($pattern, $code)) {
    die("Invalid input!");
}
```

The pattern:

```text
[A-Za-z\s]+
```

allows only:

* Uppercase letters
* Lowercase letters
* Spaces

Any other character causes the request to be rejected.

For example:

```sql
'; SELECT 1,2,3,4-- -
```

would fail validation and never reach the database.

Input validation is especially useful when the expected input format is well defined, such as:

* Email addresses
* Port codes
* Phone numbers
* ZIP codes
* Dates
* Numeric IDs

# User Privileges (Least Privilege Principle)

Applications should never connect to databases using administrative or superuser accounts.

Instead, create dedicated database users with only the permissions required for the application.

Example:

```sql
CREATE USER 'reader'@'localhost';
GRANT SELECT ON ilfreight.ports TO 'reader'@'localhost' IDENTIFIED BY 'p@ssw0Rd!!';
```

The user above can:

```sql
SELECT * FROM ports;
```

But cannot access unrelated tables:

```sql
SELECT * FROM credentials;
```

Result:

```text
ERROR 1142 (42000): SELECT command denied
```

Benefits of least privilege:

* Limits damage if SQL Injection occurs
* Prevents access to sensitive tables
* Blocks dangerous operations such as:

  * DROP
  * DELETE
  * UPDATE
  * FILE operations
  * User management

A compromised low-privilege account is far less dangerous than a compromised administrator account.

# Web Application Firewall (WAF)

A Web Application Firewall (WAF) sits between users and the web application and inspects incoming HTTP requests.

Its purpose is to detect and block malicious payloads before they reach the application.

Common WAF solutions:

* ModSecurity
* Cloudflare WAF
* AWS WAF
* Imperva
* F5 ASM

A WAF may block requests containing suspicious SQL Injection patterns such as:

```sql
UNION SELECT
```

```sql
INFORMATION_SCHEMA
```

```sql
LOAD_FILE
```

```sql
xp_cmdshell
```

Although WAFs add an additional layer of protection, they should never be considered a replacement for secure coding practices.

# Parameterized Queries (Prepared Statements)

Parameterized queries are the most effective defense against SQL Injection.

Instead of inserting user input directly into the SQL statement, placeholders are used and values are bound separately.

Example:

```php
$username = $_POST['username'];
$password = $_POST['password'];

$query = "SELECT * FROM logins WHERE username=? AND password=?";
$stmt = mysqli_prepare($conn, $query);

mysqli_stmt_bind_param($stmt, 'ss', $username, $password);
mysqli_stmt_execute($stmt);

$result = mysqli_stmt_get_result($stmt);
```

Key idea:

```sql
SELECT * FROM logins WHERE username=? AND password=?
```

The placeholders (`?`) separate SQL logic from user data.

Even if an attacker submits:

```sql
admin' OR 1=1-- -
```

the database treats it as plain text rather than executable SQL code.

Benefits of parameterized queries:

* Prevent SQL Injection completely when implemented correctly
* Separate code from data
* Supported by all major database systems
* Easier to maintain and secure

Prepared statements should always be the preferred method when interacting with databases.

---

# Defense-in-Depth Approach

No single control should be trusted on its own.

A secure application combines multiple protections:

```text
1. Input Validation
2. Input Sanitization
3. Parameterized Queries
4. Least Privilege Database Accounts
5. Web Application Firewall
6. Secure Coding Practices
7. Continuous Security Testing
```

Using several layers together significantly reduces the risk of successful SQL Injection attacks.

---

# Key CPTS Exam Takeaways

* Never trust user input.
* Sanitize and validate all external data.
* Avoid dynamic SQL whenever possible.
* Use prepared statements and parameterized queries.
* Grant only the minimum permissions required to database accounts.
* Deploy WAFs as an additional security layer.
* Remember that defense-in-depth is the most effective strategy against SQL Injection.

A secure application assumes that user input is hostile and implements multiple layers of protection to prevent SQL Injection from becoming exploitable.
