# Intro to MySQL

This section introduces the basics of MySQL and SQL, which are essential for understanding SQL Injection. Before learning how SQL Injection works, it is important to understand how databases operate, how queries are written, and how applications interact with database management systems.

MySQL and MariaDB are among the most common relational database management systems (RDBMS) used in web applications. SQL Injection vulnerabilities usually target these systems by manipulating SQL queries executed by the backend application.

---

# Structured Query Language (SQL)

Structured Query Language (SQL) is the standard language used to interact with relational databases.

Although SQL syntax may slightly differ between database systems like:
- MySQL
- MariaDB
- PostgreSQL
- MSSQL
- Oracle

all of them generally follow the ISO SQL standard.

In this module, the syntax used is:
```text
MySQL / MariaDB syntax
```

SQL is used for many operations inside a database, including:

- Retrieving data
- Updating data
- Deleting data
- Creating databases
- Creating tables
- Managing users
- Assigning permissions

---

# Connecting to MySQL from the Command Line

The `mysql` client utility is used to authenticate and interact with a MySQL/MariaDB database server.

Basic syntax:

```bash
mysql -u root -p
```

### Explanation

| Option | Purpose |
|---|---|
| `-u` | Specifies the username |
| `-p` | Prompts for password |

---

# Secure Password Usage

When using `-p`, it is recommended to leave the password blank so MySQL prompts for it securely.

Example:

```bash
mysql -u root -p
```

After running the command:

```text
Enter password:
```

This is safer because:
- Passwords are not exposed on-screen
- Passwords are not saved in bash history
- Passwords do not appear in process listings

---

# Supplying Password Directly

It is also possible to provide the password directly:

```bash
mysql -u root -ppassword
```

### Important Note

There must NOT be a space between:
```text
-p
```
and the password.

Correct:
```bash
-pmypassword
```

Incorrect:
```bash
-p mypassword
```

---

# Why Direct Passwords Are Dangerous

Using passwords directly in commands is discouraged because:
- The password may be stored in `.bash_history`
- The password may appear in logs
- Other users may see it through process monitoring tools

---

# Connecting to Remote Databases

If no host is specified, MySQL connects to:
```text
localhost
```

To connect to a remote server:

```bash
mysql -u root -h docker.hackthebox.eu -P 3306 -p
```

### Explanation

| Option | Purpose |
|---|---|
| `-h` | Target host |
| `-P` | Port number |
| `-p` | Prompt for password |

---

# MySQL Default Port

Default MySQL/MariaDB port:
```text
3306
```

### Important Difference

| Flag | Purpose |
|---|---|
| `-p` | Password |
| `-P` | Port |

Lowercase `p` is for password.

Uppercase `P` is for port.

---

# SQL Case Sensitivity

SQL keywords are generally:
```text
Case-insensitive
```

These are equivalent:

```sql
USE users;
```

```sql
use users;
```

However:
```text
Database names may be case-sensitive
```

Example:

```sql
USE USERS;
```

may fail if the actual database name is:
```text
users
```

Best practice:
- Write SQL keywords in uppercase
- Keep database/table names consistent

# Tables in Databases

Databases store information inside:
```text
Tables
```

A table consists of:
- Rows
- Columns

### Important Concepts

| Term | Meaning |
|---|---|
| Row | A single record |
| Column | A field/category |
| Cell | Intersection of row and column |

---

# Data Types

Every column must have a data type that defines what kind of information it stores.

Common data types:

| Data Type | Purpose |
|---|---|
| `INT` | Integer numbers |
| `VARCHAR` | Variable-length strings |
| `DATETIME` | Date and time |
| `DATE` | Date only |

---

# Creating a Table

Example table:

```sql
CREATE TABLE logins (
    id INT,
    username VARCHAR(100),
    password VARCHAR(100),
    date_of_joining DATETIME
);
```

---

# Understanding the Table Structure

This table creates four columns:

| Column | Data Type | Purpose |
|---|---|
| `id` | INT | Stores numeric IDs |
| `username` | VARCHAR(100) | Stores usernames |
| `password` | VARCHAR(100) | Stores passwords |
| `date_of_joining` | DATETIME | Stores registration date |

---

# VARCHAR Explained

```sql
VARCHAR(100)
```

means:
```text
Variable-length string up to 100 characters
```

If input exceeds 100 characters:
- MySQL may reject the data
- Or truncate it depending on configuration

# Table Properties and Constraints

Columns can have additional rules and properties.

These are called:
```text
Constraints
```

They control:
- Data validation
- Integrity
- Relationships

---

# AUTO_INCREMENT

```sql
id INT NOT NULL AUTO_INCREMENT
```

This automatically increments the value every time a new row is inserted.

Example:

| Insert | Generated ID |
|---|---|
| First row | 1 |
| Second row | 2 |
| Third row | 3 |

---

# NOT NULL

```sql
username VARCHAR(100) NOT NULL
```

This means:
```text
The field cannot be empty
```

It becomes a required field.

---

# UNIQUE Constraint

```sql
username VARCHAR(100) UNIQUE
```

Ensures:
```text
No duplicate values are allowed
```

Example:
- Two users cannot share the same username

---

# DEFAULT Values

```sql
date_of_joining DATETIME DEFAULT NOW()
```

If no value is provided:
- MySQL automatically inserts the current date and time

---

# PRIMARY KEY

```sql
PRIMARY KEY (id)
```

The primary key uniquely identifies every record in the table.

Important characteristics:
- Must be unique
- Cannot be NULL
- Commonly used for relationships between tables

---

# Final Secure Table Example

```sql
CREATE TABLE logins (
    id INT NOT NULL AUTO_INCREMENT,
    username VARCHAR(100) UNIQUE NOT NULL,
    password VARCHAR(100) NOT NULL,
    date_of_joining DATETIME DEFAULT NOW(),
    PRIMARY KEY (id)
);
```

---

# Understanding the Final Table

This table now includes:
- Automatic ID generation
- Unique usernames
- Required username/password fields
- Automatic timestamps
- Primary key enforcement

This structure is much closer to real-world production databases.

---

# Why These Concepts Matter for SQL Injection

Understanding:
- Tables
- Columns
- Data types
- SQL syntax
- Constraints
- Database interaction

is critical before learning SQL Injection because attackers manipulate SQL queries directly.

To exploit SQL Injection successfully, a penetration tester must understand:
- How queries are structured
- How data is stored
- How authentication systems query databases
- How backend applications interact with SQL servers

Without understanding SQL fundamentals, SQL Injection becomes difficult to analyze and exploit properly.
