## 📑 CPTS Exam Study Notes: Union-Based SQL Injection

### 1. Fundamentals of Union-Based SQLi

* **Prerequisites:** * The application must be vulnerable to SQL injection (identifiable by triggering a syntax error using characters like a single quote `'`).
* The application must display the query results directly on the web page (**In-band SQLi**).


* **Objective:** Merge the results of the original application query with a malicious secondary query to extract data from other tables or databases.

---

### 2. Phase 1: Detecting the Number of Columns

To use the `UNION` operator, the injected query **must have the exact same number of columns** as the original query. There are two primary techniques to determine this count:

#### Method A: Using `ORDER BY`

* **Concept:** The `ORDER BY` clause sorts results by a specified column index. If you sort by an index higher than the total number of columns available, the database throws an error.
* **Approach:** Start at 1 and increment sequentially (`ORDER BY 1`, `ORDER BY 2`, etc.) until the page breaks or throws an "Unknown column" error.
* **Exam Payload Pattern:**
```sql
' ORDER BY 1-- -
' ORDER BY 5-- -  -- [Fails: Indicates exactly 4 columns exist]

```



#### Method B: Using `UNION SELECT`

* **Concept:** Manually injecting a `UNION SELECT` statement with a varying number of dummy values. If the column count does not match the original query, the database rejects the statement with an error.
* **Approach:** Increment the number of arguments until the application successfully processes the request.
* **Exam Payload Pattern:**
```sql
cn' UNION SELECT 1,2,3-- -     -- [Error: Column count mismatch]
cn' UNION SELECT 1,2,3,4-- -   -- [Success: Indicates exactly 4 columns exist]

```



---

### 3. Phase 2: Identifying Reflected / Displayed Columns

Even if a query returns multiple columns (e.g., 4 columns), the web application's UI might only print/render a subset of them to the screen.

* **Why it matters:** If you place an exploit payload (e.g., extracting passwords) into a column that is hidden from the user interface, you will receive no output.
* **Detection Technique:** Use sequential numbers (`1, 2, 3, 4`) as junk/dummy data in your `UNION SELECT` payload. Look at the web page output to see which numbers actually appear in the UI.
* **Example Analysis:**
* **Payload:** `cn' UNION SELECT 1,2,3,4-- -`
* **Result:** The UI displays the data positions for `2`, `3`, and `4`, but **omits `1**`.
* **Takeaway:** Columns 2, 3, and 4 are valid vectors for data extraction. Column 1 is dead/unusable for information gathering.



---

### 4. Phase 3: Initial PoC & Enumeration Testing

Once the usable column positions are mapped out, replace one of the reflected dummy numbers with a built-in database function or variable to confirm data retrieval.

* **Test Payload (MySQL/MariaDB Fingerprinting):**
```sql
cn' UNION SELECT 1, @@version, 3, 4-- -

```


* **Expected Behavior:** The position previously holding the number `2` will now dynamically display the active backend environment string (e.g., `10.3.22-MariaDB-1ubuntu1`).

---

### 💡 CPTS Exam Cheat Sheet Tips

1. **The Space Character (`-- -`):** Always remember the trailing space after the SQL comment indicator (`--`). In web requests, a trailing space can easily be dropped or ignored by browsers, so typing it explicitly or URL-encoding it as `--+` or `--%20` is vital for breaking the rest of the backend application's original query safely.
2. **Fingerprinting:** Seeing a response containing `MariaDB` or `Ubuntu` allows you to immediately tailor your subsequent enumeration scripts/payloads strictly to MySQL/MariaDB syntax, saving valuable exam time.
