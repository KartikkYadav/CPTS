# Stored XSS (Persistent XSS) - CPTS Notes

## Introduction

Stored XSS, also known as **Persistent XSS**, is the most dangerous and impactful type of Cross-Site Scripting vulnerability. Unlike Reflected XSS, where the payload only exists for a single request and response cycle, Stored XSS allows the attacker's payload to be permanently saved by the application.

The injected payload is typically stored in a backend database and automatically retrieved whenever users access the affected page. Because the malicious code remains stored on the server, every user who visits the vulnerable page becomes a potential victim.

This persistence makes Stored XSS significantly more dangerous than other XSS types because a single successful injection can affect hundreds or even thousands of users without requiring any further interaction from the attacker.

---

# How Stored XSS Works

A typical Stored XSS attack follows this workflow:

```text
Attacker Submits Malicious Input
            ↓
Application Stores Input In Database
            ↓
Victim Visits Page
            ↓
Application Retrieves Stored Data
            ↓
Browser Renders Page
            ↓
Malicious JavaScript Executes
```

Unlike Reflected XSS, the victim does not need to click a specially crafted URL. Simply visiting the affected page is enough to trigger the attack.

---

# Example Scenario

Consider a simple To-Do List application where users can add tasks.

Normal Input:

```text
Buy groceries
```

The application stores the task and later displays it:

```html
<ul>
  <li>Buy groceries</li>
</ul>
```

Everything works normally.

However, if the application fails to sanitize user input, an attacker may submit:

```html
<script>alert(window.origin)</script>
```

The application stores this payload inside the database.

Whenever the page loads, the application retrieves the stored content and generates:

```html
<ul>
  <script>alert(window.origin)</script>
</ul>
```

The browser interprets the `<script>` tag and executes the JavaScript code.

---

# Basic XSS Testing Payload

The most common payload used to verify XSS vulnerabilities is:

```html
<script>alert(window.origin)</script>
```

### Why Use `window.origin`?

Many applications use:

```javascript
alert(1)
```

for testing.

However:

```javascript
alert(window.origin)
```

is more useful because it displays the exact domain where the payload executes.

Example:

```text
http://vulnerable-site.com
```

This helps determine:

* Which page is vulnerable
* Which domain is executing the payload
* Whether an iframe is involved

---

# Verifying the Vulnerability

After submitting the payload:

```html
<script>alert(window.origin)</script>
```

A popup appears displaying:

```text
http://vulnerable-site.com
```

This confirms that:

1. User input reached the page.
2. The browser interpreted it as HTML.
3. JavaScript execution was successful.
4. The application is vulnerable to XSS.

---

# Viewing the Page Source

To verify the payload is actually present inside the page:

```text
CTRL + U
```

or

```text
Right Click → View Page Source
```

Example:

```html
<div></div>
<ul class="list-unstyled" id="todo">
    <ul>
        <script>alert(window.origin)</script>
    </ul>
</ul>
```

The payload is clearly embedded inside the HTML response.

---

# Alternative XSS Testing Payloads

Some browsers or environments may block `alert()`.

Because of this, it is useful to know alternative payloads.

---

## Plaintext Payload

```html
<plaintext>
```

This payload stops the browser from rendering subsequent HTML elements and displays everything as plain text.

This can be a useful visual indicator of injection.

---

## Print Dialog Payload

```html
<script>print()</script>
```

Instead of displaying an alert box, it opens the browser's print dialog.

Advantages:

* Easy to notice
* Rarely blocked by browsers
* Confirms JavaScript execution

# Determining Persistence

The key difference between Stored XSS and Reflected XSS is persistence.

To verify persistence:

### Step 1

Inject the payload:

```html
<script>alert(window.origin)</script>
```

### Step 2

Observe the alert.

### Step 3

Refresh the page.

### Step 4

If the alert appears again:

```text
Stored XSS Confirmed
```

The payload survived the refresh because it was stored in the backend database.

---

# Why Stored XSS Is Dangerous

With Reflected XSS:

```text
Victim Must Click Link
```

With Stored XSS:

```text
Victim Only Needs To Visit Page
```

This dramatically increases the number of affected users.

Examples:

* Blog comments
* Forum posts
* User profiles
* Product reviews
* Chat applications
* Ticketing systems
* Social media posts

A single payload may automatically attack every visitor.

---

# Real-World Impact

Stored XSS can be used to:

### Session Hijacking

```javascript
document.cookie
```

Steal authentication cookies and hijack accounts.

---

### Account Takeover

Perform actions using the victim's authenticated session.

Examples:

* Change password
* Change email
* Add attacker-controlled accounts

---

### Credential Theft

Inject fake login forms and capture credentials.

---

### Phishing

Redirect users to attacker-controlled websites.

---

### Malware Distribution

Load malicious JavaScript from external servers.

---

### Worm Propagation

Stored XSS can self-replicate.

The most famous example is:

```text
Samy Worm (MySpace - 2005)
```

which infected over one million users within a day.

---

# Stored XSS vs Reflected XSS

| Feature                    | Stored XSS | Reflected XSS |
| -------------------------- | ---------- | ------------- |
| Stored in Database         | Yes        | No            |
| Survives Refresh           | Yes        | No            |
| Affects Multiple Users     | Yes        | Usually No    |
| Requires Victim Link Click | Usually No | Usually Yes   |
| Severity                   | High       | Medium        |

---

# Common Locations to Test

Stored XSS is commonly found in:

* Comments
* Reviews
* Support Tickets
* User Profiles
* Chat Messages
* Forum Posts
* Contact Forms
* Feedback Sections
* Social Media Posts

Any location where user input is saved should be tested.

---

# Detection Methodology

When testing for Stored XSS:

```text
1. Identify Input Field
2. Submit XSS Payload
3. Observe Response
4. Refresh Page
5. Revisit Page
6. Log Out and Revisit
7. Check Different User Accounts
8. Verify Persistent Execution
```

If the payload executes for other users or after page refreshes, it is likely Stored XSS.

# CPTS Exam Notes

### Key Characteristics

```text
Payload Stored In Database
          ↓
Retrieved Later
          ↓
Executed Automatically
```

### Remember

```text
Stored XSS = Persistent XSS
```

### Indicators

* Payload survives page refresh
* Payload executes for all visitors
* Payload remains until manually removed

### Severity

```text
Stored XSS
=
Most Dangerous XSS Type
```

because a single injection can compromise every user who visits the vulnerable page.
