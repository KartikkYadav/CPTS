# Introduction

As web applications continue to become more advanced and widely used, web application vulnerabilities also continue to grow. One of the most common vulnerabilities encountered during web penetration testing is **Cross-Site Scripting (XSS)**.

XSS occurs when a web application fails to properly sanitize or validate user-controlled input before displaying it inside a web page. Because of this, an attacker can inject JavaScript code that will execute inside another user's browser.

Unlike many server-side vulnerabilities, XSS is primarily a **client-side vulnerability**. The malicious code runs inside the victim's browser rather than directly on the web server.

---

# What is XSS?

A normal web application works as follows:

```text
User Request
     ↓
Web Server
     ↓
HTML Response
     ↓
Browser Renders Page
```

When user input is not properly sanitized:

```text
Attacker Input
     ↓
Application Stores/Processes Input
     ↓
Malicious JavaScript Returned
     ↓
Victim Browser Executes Script
```

The attacker injects JavaScript code into an application input field such as:

* Comments
* Search boxes
* Contact forms
* User profiles
* Feedback forms
* Messages

When another user loads the page, the browser interprets the injected JavaScript as legitimate code and executes it.

---

# Important Characteristics of XSS

### Client-Side Vulnerability

XSS executes entirely inside the victim's browser.

The web server is not directly compromised.

---

### Impacts Users Instead of Servers

The attacker targets the user viewing the page.

The browser executes the attacker's JavaScript with the permissions of the vulnerable website.

---

### Extremely Common

XSS vulnerabilities have existed for over 20 years and continue to appear in modern web applications.

Even major organizations such as:

* Google
* Twitter
* Apache
* MySpace

have suffered from XSS vulnerabilities.

---

# Risk of XSS

Although XSS generally has a lower direct impact on the backend server, it occurs very frequently.

Risk can be viewed as:

```text
Risk = Impact × Probability
```

For XSS:

```text
Low/Medium Impact
+
High Probability
=
Medium Risk
```

Because of its frequency, XSS should always be taken seriously during penetration tests.

---

# What Can XSS Be Used For?

Anything that can be performed using JavaScript inside a browser can potentially be performed through XSS.

Common attacks include:

---

## Session Cookie Theft

The attacker steals authentication cookies.

```javascript
document.cookie
```

Stolen session cookies may allow account takeover.

---

## Account Takeover

Instead of stealing cookies, attackers may perform actions as the victim.

Examples:

* Change password
* Change email address
* Add attacker-controlled accounts

---

## Unauthorized API Requests

JavaScript can send requests to application APIs using the victim's authenticated session.

Examples:

```text
Transfer money
Delete records
Modify profile settings
Create new users
```

---

## Credential Theft

Attackers can inject fake login forms to capture usernames and passwords.

---

## Phishing

Victims can be redirected to attacker-controlled websites.

---

## Displaying Advertisements

Malicious JavaScript can inject unwanted content or advertisements.

---

## Cryptocurrency Mining

Browser resources can be abused for crypto mining operations.

---

# Browser Security Limitations

Although powerful, XSS has limitations.

JavaScript executes within the browser sandbox.

Normally it cannot:

```text
Execute OS Commands
Read Local Files
Gain Direct Server Access
```

The code is generally restricted to the current website's domain.

However, this restriction still allows attackers to perform many dangerous actions on behalf of users.

---

# Browser Exploitation Possibilities

A highly skilled attacker may combine:

```text
XSS
+
Browser Vulnerability
```

For example:

```text
XSS
     ↓
Browser Memory Corruption Bug
     ↓
Sandbox Escape
     ↓
Code Execution on Victim Machine
```

While uncommon, this demonstrates why XSS vulnerabilities should never be ignored.

# Famous Real-World XSS Cases

## Samy Worm (MySpace - 2005)

One of the most famous XSS worms ever created.

The payload posted:

```text
"Samy is my hero"
```

on victim profiles.

Every infected profile automatically infected visitors viewing it.

Results:

```text
Over 1 Million Infected Profiles
Within 24 Hours
```

Although mostly harmless, the same vulnerability could have been used for:

* Credential theft
* Credit card theft
* Browser exploitation
* Malware distribution

---

## Twitter TweetDeck XSS (2014)

A researcher discovered an XSS vulnerability inside TweetDeck.

The payload automatically retweeted itself.

Results:

```text
38,000+ Retweets
In Less Than 2 Minutes
```

Twitter temporarily shut down TweetDeck while patching the vulnerability.

---

## Google XSS Vulnerabilities

Google Search has experienced multiple XSS vulnerabilities over the years.

One notable case involved an issue in an XML processing library discovered in 2019.

---

## Apache XSS Vulnerability

Apache also reported XSS vulnerabilities that attackers actively exploited to steal user credentials.

# Types of XSS

There are three primary XSS categories.

| Type                           | Description                                                                         |
| ------------------------------ | ----------------------------------------------------------------------------------- |
| Stored (Persistent) XSS        | Payload is stored in a database and executes whenever users view the page           |
| Reflected (Non-Persistent) XSS | Payload is reflected immediately in the response but not stored                     |
| DOM-Based XSS                  | Payload is processed entirely by client-side JavaScript without reaching the server |

---

# Stored (Persistent) XSS

Stored XSS is the most dangerous form of XSS.

### How It Works

```text
Attacker Submits Payload
          ↓
Payload Stored In Database
          ↓
Victim Opens Page
          ↓
Payload Retrieved
          ↓
Browser Executes Script
```

Common locations:

* Blog comments
* Forum posts
* User profiles
* Reviews
* Chat messages

Because the payload is stored permanently, every visitor may be affected.

---

# Reflected (Non-Persistent) XSS

Reflected XSS occurs when input is:

```text
Sent To Server
     ↓
Processed
     ↓
Returned To User
     ↓
Executed By Browser
```

The payload is not stored.

Common examples:

* Search results
* Error messages
* Confirmation messages
* URL parameters

Victims are usually tricked into clicking malicious links.

---

# DOM-Based XSS

DOM-Based XSS occurs entirely inside the browser.

The payload never reaches the backend server.

Flow:

```text
User Input
     ↓
Client-Side JavaScript
     ↓
DOM Modification
     ↓
JavaScript Execution
```

Common sources:

* URL fragments (#)
* Client-side parameters
* JavaScript frameworks
* Dynamic page updates

---

# XSS Quick Comparison

| Feature                    | Stored XSS | Reflected XSS | DOM XSS   |
| -------------------------- | ---------- | ------------- | --------- |
| Stored in Database         | Yes        | No            | No        |
| Requires Victim Link Click | Sometimes  | Usually       | Usually   |
| Server Involved            | Yes        | Yes           | No        |
| Persistence                | Permanent  | Temporary     | Temporary |
| Severity                   | Highest    | Medium        | Medium    |

# CPTS Exam Notes

### Remember

```text
Stored XSS
=
Payload Stored
=
Most Dangerous
```

```text
Reflected XSS
=
Payload Reflected
=
Not Stored
```

```text
DOM XSS
=
Processed By Browser
=
Never Reaches Server
```

### Common Impacts

* Session Hijacking
* Credential Theft
* Account Takeover
* CSRF-like Actions
* Phishing
* Browser Exploitation
* Malicious Redirects

### Key Exam Point

XSS is a **client-side vulnerability** that executes JavaScript within the victim's browser. While it may not directly compromise the server, it can completely compromise user accounts and perform actions on behalf of authenticated users.
