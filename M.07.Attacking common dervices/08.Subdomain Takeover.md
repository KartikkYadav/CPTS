# Latest DNS Vulnerabilities: Subdomain Takeover

---

We can find thousands of subdomains and domains on the web that point to no longer active third-party service providers such as AWS, GitHub, and others. At best, they display an error message as confirmation of a deactivated third-party service.

* **The Root Cause:** Large companies and corporations are affected time and again. Companies often cancel services from third-party providers but forget to delete the associated DNS records because no additional costs are incurred for keeping an idle DNS entry.
* **Bug Bounty Impact:** Many well-known bug bounty platforms, such as **HackerOne**, explicitly list **Subdomain Takeover** as a high-reward bounty category. Dozens of tools on GitHub automate the discovery of vulnerable subdomains or help create Proof of Concepts (PoC).
* **The Scale:** A 2020 study by RedHuntLabs found that over **400,000 subdomains** out of 220 million analyzed were vulnerable to subdomain takeover. Notably, **62%** of these vulnerable domains belonged to the e-commerce sector.

---

## The Concept of the Attack

One of the biggest dangers of a subdomain takeover is that a phishing campaign can be launched that appears to be part of the official, trusted domain of the target company.

For example, a customer inspecting a link would see that the domain `customer-drive.inlanefreight.com` (which points to a non-existent S3 bucket from AWS) sits directly behind the official company domain `inlanefreight.com` and trust it completely. The customer has no way of knowing that this page has been mirrored or created by an attacker to harvest corporate credentials.

### The Technical Vulnerability Triangle

If an attacker finds a **CNAME** record in the company's DNS records that points to a subdomain that no longer exists and returns an **HTTP 404 error** (or similar provider-specific error message), this subdomain can most likely be taken over.

1. A subdomain points to an external third-party domain using a CNAME record.
2. The external domain or service asset space currently does not exist (it was deleted or expired).
3. When an attacker registers this non-existent resource name on the provider's platform, the legitimate corporate subdomain points directly to the attacker's space.

By claiming that vacant asset space on the third-party provider, the attacker effectively becomes the owner of that particular subdomain's web traffic without modifying the target company's DNS server directly.

## Initiation of Subdomain Takeover

The initial phase focuses on discovering the dangling record and claiming the vacant third-party asset space to establish ownership:

| Step | Subdomain Takeover Attack Flow | Category |
| --- | --- | --- |
| **1.** | The source is the dangling subdomain name that is no longer used by the company but was discovered active in their DNS records. | **Source** |
| **2.** | The registration of this asset on the third-party provider's site (e.g., AWS, GitHub) is completed by linking it to our own custom malicious source. | **Process** |
| **3.** | The privileges lie with the primary domain owner and its entries in its DNS servers. The third-party provider is not responsible for validating whether this subdomain should be accessible. | **Privileges** |
| **4.** | The successful registration and mapping link directly to our server, which acts as the destination in this phase. | **Destination** |

---

## Triggering the Traffic Forwarding

Once the vacant resource is claimed, the operational cycle relies on standard internet infrastructure to route unsuspecting users to the attacker-controlled environment:

| Step | Subdomain Takeover Attack Flow | Category |
| --- | --- | --- |
| **5.** | The visitor enters the legitimate company URL in their browser, using the outdated, unremoved DNS record (CNAME) as the source. | **Source** |
| **6.** | The authoritative DNS server looks up the subdomain, sees the valid CNAME pointer, and seamlessly redirects the user to the corresponding resource (now controlled by us). | **Process** |
| **7.** | The trust privileges lie with the original domain administrators; because this subdomain is explicitly allowed in their records, local DNS servers view it as trustworthy and forward the visitor. | **Privileges** |
| **8.** | The destination is the client browser or machine that requests the IP address of the subdomain and gets forwarded directly to our malicious infrastructure. | **Destination** |

## Beyond Phishing: Secondary Attack Vectors

Subdomain takeover is a devastating exploit because the malicious page runs under the exact same parent domain space as the primary target organization. This inheritance allows attackers to execute a variety of secondary web attacks:

* **Session Cookie Stealing:** Since browsers often scope session cookies to the parent domain (e.g., `*.inlanefreight.com`), the attacker's hijacked subdomain can read sensitive session tokens belonging to users visiting the main site.
* **Cross-Site Request Forgery (CSRF):** Attackers can bypass multi-layered CSRF protections that inherently trust requests coming from any subdomain of the core site.
* **Abusing CORS (Cross-Origin Resource Sharing):** If the company's internal APIs utilize relaxed CORS configurations (such as trusting `*.inlanefreight.com`), the hijacked subdomain can steal sensitive data directly from authenticated users via API calls.
* **Defeating Content Security Policy (CSP):** Modern web applications restrict script execution using CSP white-lists. If the company's security policy explicitly trusts its own subdomains, the attacker can bypass CSP barriers to execute malicious scripts natively.
