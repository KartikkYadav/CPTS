# SQLMap Requests & Input Handling (CPTS Notes)

One of the most common reasons SQLMap fails to find a SQL Injection vulnerability is not because the application is secure, but because the request provided to SQLMap is incomplete or incorrect.

Missing cookies, incorrect POST data, missing headers, CSRF tokens, or using the wrong HTTP method can all prevent SQLMap from properly testing a target.

Before running SQLMap, always make sure the request you provide is identical to the request that successfully reaches the application.

---

# Using Browser Requests with SQLMap

The easiest way to create a proper SQLMap command is by using the browser's **Copy as cURL** feature.

### Steps

1. Open Developer Tools (`F12`)
2. Go to the **Network** tab
3. Find the target request
4. Right-click it
5. Select:

```text
Copy → Copy as cURL
```

Example:

```bash
curl 'http://www.example.com/?id=1' \
-H 'User-Agent: Mozilla/5.0' \
-H 'Accept: image/webp,*/*'
```

Simply replace:

```bash
curl
```

with:

```bash
sqlmap
```

Example:

```bash
sqlmap 'http://www.example.com/?id=1' \
-H 'User-Agent: Mozilla/5.0'
```

This ensures SQLMap sends exactly the same request as your browser.

---

# Testing GET Requests

GET parameters are usually passed through the URL.

Example:

```text
http://www.example.com/?id=1
```

SQLMap:

```bash
sqlmap -u "http://www.example.com/?id=1"
```

SQLMap automatically tests:

```text
id
```

for SQL Injection.

---

# Testing POST Requests

Many login forms and APIs use POST requests.

Example POST body:

```text
uid=1&name=test
```

SQLMap:

```bash
sqlmap -u "http://www.example.com/" \
--data="uid=1&name=test"
```

SQLMap will test:

```text
uid
name
```

for SQL Injection.

---

# Testing a Specific Parameter

If you already know which parameter is interesting, use:

```bash
-p
```

Example:

```bash
sqlmap -u "http://www.example.com/" \
--data="uid=1&name=test" \
-p uid
```

Only:

```text
uid
```

will be tested.

---

# Using the Injection Marker (*)

Instead of testing every parameter, SQLMap can be told exactly where to inject.

Example:

```bash
sqlmap -u "http://www.example.com/" \
--data="uid=1*&name=test"
```

The asterisk tells SQLMap:

```text
Inject here.
```

This reduces unnecessary testing.

---

# Full HTTP Requests (-r)

For complex applications, URLs and POST data are often not enough.

Examples:

- Custom Headers
- Authentication Tokens
- Session Cookies
- API Requests
- Large JSON Bodies

In these situations, save the entire request into a file.

Example:

```http
GET /?id=1 HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Cookie: PHPSESSID=abcd1234
```

Save as:

```text
req.txt
```

Run:

```bash
sqlmap -r req.txt
```

SQLMap will replay the exact request.

---

# Using * Inside Request Files

You can also specify the injection point.

Example:

```http
GET /?id=* HTTP/1.1
Host: www.example.com
```

SQLMap will only test:

```text
id
```

at the marked location.

---

# Using Cookies

Authenticated pages often require valid session cookies.

Example:

```text
PHPSESSID=ab4530f4a7d10448457fa8b0eadac29c
```

Provide it with:

```bash
sqlmap --cookie="PHPSESSID=ab4530f4a7d10448457fa8b0eadac29c"
```

Alternative:

```bash
sqlmap -H "Cookie: PHPSESSID=ab4530f4a7d10448457fa8b0eadac29c"
```

Without valid cookies SQLMap may only see:

```text
Login Page
```

instead of the vulnerable functionality.

---

# Custom Headers

Headers can be manually supplied.

Example:

```bash
-H "Referer: https://target.com"
```

```bash
-H "Host: target.com"
```

```bash
-H "X-Forwarded-For: 127.0.0.1"
```

```bash
-A "Mozilla/5.0"
```

Useful when the application requires specific headers.

---

# Random User-Agent

Many WAFs and security solutions block SQLMap's default User-Agent.

Use:

```bash
--random-agent
```

Example:

```bash
sqlmap -u http://target.com --random-agent
```

SQLMap will randomly choose a legitimate browser User-Agent.

Example:

```text
Chrome
Firefox
Safari
Edge
```

---

# Mobile User-Agent

To imitate mobile traffic:

```bash
--mobile
```

Example:

```bash
sqlmap -u http://target.com --mobile
```

Useful when applications behave differently for mobile users.

---

# Testing Headers for SQL Injection

SQL Injection is not limited to URL parameters.

Headers can also be vulnerable.

Example:

```bash
--cookie="id=1*"
```

Injection point:

```text
*
```

SQLMap will test inside the cookie value.

This can also be done with:

```text
Referer
Host
User-Agent
X-Forwarded-For
```

and other headers.

---

# Using Custom HTTP Methods

Default methods:

```text
GET
POST
```

Some APIs use:

```text
PUT
PATCH
DELETE
```

Example:

```bash
sqlmap -u http://target.com \
--data="id=1" \
--method PUT
```

SQLMap will use:

```http
PUT
```

instead of POST.

---

# JSON Requests

Modern APIs frequently use JSON.

Example:

```json
{
  "id": 1,
  "name": "admin"
}
```

SQLMap supports JSON automatically.

Example:

```bash
sqlmap -u http://target.com/api \
--data='{"id":1}'
```

When SQLMap detects JSON:

```text
JSON data found in HTTP body.
Do you want to process it?
```

Select:

```text
Y
```

---

# Complex JSON Requests

For large API requests, save the entire request.

Example:

```http
POST /api/articles HTTP/1.1
Host: target.com

{
  "data": [{
      "id": "1",
      "title": "Example"
  }]
}
```

Save as:

```text
req.txt
```

Run:

```bash
sqlmap -r req.txt
```

This is usually the preferred approach during API testing.

---

# XML Requests

SQLMap also supports XML.

Example:

```xml
<user>
    <id>1</id>
</user>
```

Run:

```bash
sqlmap --data="<user><id>1</id></user>"
```

SQLMap automatically parses and tests XML parameters.

# CPTS Exam Tips

### Always Use Burp

Capture the request first.

Then:

```text
Right Click
→ Save Item
```

or

```text
Copy Request
```

Use:

```bash
sqlmap -r request.txt
```

This avoids missing:

- Cookies
- Headers
- Tokens
- JSON fields
- Authentication data

---

### Most Important Options

| Option | Purpose |
|----------|----------|
| `-u` | Target URL |
| `--data` | POST data |
| `-p` | Specific parameter |
| `-r` | Full HTTP request |
| `--cookie` | Session cookie |
| `-H` | Custom header |
| `--random-agent` | Random browser User-Agent |
| `--mobile` | Mobile User-Agent |
| `--method` | Custom HTTP method |
| `*` | Injection marker |

---

# Quick Rule for CPTS

```text
Simple GET Request
        ↓
Use -u

POST Form
        ↓
Use --data

Authenticated Application
        ↓
Use --cookie

Complex Request/API
        ↓
Use -r

Known Injection Point
        ↓
Use *

Blocked by WAF
        ↓
Use --random-agent
```

The most reliable and exam-friendly approach is:

```bash
sqlmap -r request.txt
```

because it reproduces the exact request sent by the browser or Burp and greatly increases the chances of successful SQL Injection detection and exploitation.
