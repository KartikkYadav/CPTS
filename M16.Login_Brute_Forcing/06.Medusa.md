# Medusa

Medusa is a fast, modular, and massively parallel login brute-forcing tool widely used during:
- Penetration testing
- Red team engagements
- Credential auditing
- Weak password assessments

It is designed to test authentication services by systematically attempting multiple username and password combinations against remote services.

Unlike manual login attempts, Medusa automates the authentication process and can run:
```text
Multiple parallel login attempts simultaneously
```

which makes brute-force attacks significantly faster and more efficient. :contentReference[oaicite:0]{index=0}

---

# Why Medusa is Important in CPTS

During penetration tests, authentication services are extremely common targets because organizations frequently use:
- Weak passwords
- Default credentials
- Reused credentials
- Empty passwords
- Poor password policies

Medusa helps identify these weaknesses by automating login attacks against:
- SSH
- FTP
- HTTP
- Databases
- RDP
- Email services
- Web applications

and many other authentication protocols. :contentReference[oaicite:1]{index=1}

# Basic Medusa Syntax

```bash
medusa [target_options] [credential_options] -M module [module_options]
```

The syntax is divided into:
- Target information
- Credentials
- Service module
- Additional module options

---

# Important Medusa Parameters

| Parameter | Description |
|-----------|-------------|
| `-h HOST` | Single target host |
| `-H FILE` | File containing multiple targets |
| `-u USERNAME` | Single username |
| `-U FILE` | Username wordlist |
| `-p PASSWORD` | Single password |
| `-P FILE` | Password wordlist |
| `-M MODULE` | Service module |
| `-m` | Additional module options |
| `-t TASKS` | Number of parallel tasks |
| `-f` | Stop after first success on current host |
| `-F` | Stop after first success globally |
| `-n PORT` | Custom port |
| `-v LEVEL` | Verbose output level |

Modules are the core of Medusa.

Each module is specifically designed to communicate with a particular service or protocol.

For example:
- SSH authentication differs from FTP

Medusa modules handle these differences automatically.

# SSH Brute Force Example

Suppose we want to test an SSH server:

```text
192.168.0.100
```

We have:
- `usernames.txt`
- `passwords.txt`

---

# SSH Attack Command

```bash
medusa -h 192.168.0.100 -U usernames.txt -P passwords.txt -M ssh
```

---

# Understanding the Command

## Target Host

```bash
-h 192.168.0.100
```

Specifies:
```text
Target SSH server
```

---

## Username Wordlist

```bash
-U usernames.txt
```

Loads:
```text
List of usernames
```

---

## Password Wordlist

```bash
-P passwords.txt
```

Loads:
```text
List of passwords
```

---

## SSH Module

```bash
-M ssh
```

Tells Medusa:
```text
Attack the SSH authentication service
```

---

# What Medusa Does Internally

Medusa systematically:
1. Takes a username
2. Tries every password
3. Sends authentication requests
4. Analyzes server responses
5. Detects valid credentials

This process continues until:
- Valid credentials are discovered
- Wordlists are exhausted

# Web Form Brute Forcing

Medusa can also brute-force:
```text
Custom web login forms
```

using:
```text
web-form module
```

---

# Example

```bash
medusa -M web-form -h www.example.com \
-U users.txt -P passwords.txt \
-m FORM:"username=^USER^&password=^PASS^:F=Invalid"
```

---

# Understanding the FORM String

## ^USER^

Replaced dynamically with:
```text
Usernames
```

---

## ^PASS^

Replaced dynamically with:
```text
Passwords
```

---

## F=Invalid

Failure condition.

If:
```text
Invalid
```

appears in the response:
```text
Login failed
```

Otherwise:
```text
Potential success detected
```
