# Login Forms

Beyond Basic HTTP Authentication, most modern web applications use:
```text
Custom login forms
```

for user authentication.

These login forms may visually look different from one website to another, but internally they usually work in a very similar way.

Because login forms rely on:
- Username fields
- Password fields
- HTTP requests

they are common targets for:
- Brute-force attacks
- Password spraying
- Credential stuffing

during penetration testing and real-world attacks.

---

# Understanding Login Forms

A login form is essentially:
```text
An HTML form
```

embedded inside a webpage.

Its purpose is to:
- Collect user credentials
- Send them to the server
- Authenticate the user

Most login forms contain:
- Username input field
- Password input field
- Submit button

---

# Basic Login Form Example

```html
<form action="/login" method="post">

  <label for="username">Username:</label>

  <input type="text" id="username" name="username">

  <label for="password">Password:</label>

  <input type="password" id="password" name="password">

  <input type="submit" value="Submit">

</form>
```

---

# Understanding the HTML Form

## action="/login"

This specifies:
```text
Where the form data will be sent
```

after submission.

In this case:
```text
/login
```

is the endpoint responsible for authentication.

---

## method="post"

This specifies:
```text
HTTP POST requests
```

will be used to send credentials.

POST requests send data:
- Inside the request body
- Not in the URL

which is more secure than GET requests for authentication.

---

## Input Fields

### Username Field

```html
<input type="text" name="username">
```

The important part for attackers is:
```text
name="username"
```

because this becomes:
```text
The parameter name
```

inside the HTTP request.

---

### Password Field

```html
<input type="password" name="password">
```

Again, the important part is:
```text
name="password"
```

which identifies the password parameter.

---

# HTTP POST Request Example

When the user submits the form, the browser creates an HTTP POST request.

Example:

```http
POST /login HTTP/1.1
Host: www.example.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 29

username=john&password=secret123
```

---

# Understanding the Request

## POST /login

This tells the server:
```text
A login request is being sent
```

to:
```text
/login
```

---

## Content-Type

```http
Content-Type: application/x-www-form-urlencoded
```

This specifies:
```text
How form data is encoded
```

inside the request body.

---

## Request Body

```text
username=john&password=secret123
```

This contains:
- Username parameter
- Password parameter

formatted as:
```text
key=value pairs
```

---

# How Browsers Handle Login Forms

When a user interacts with a login form:
1. The browser captures the credentials
2. JavaScript may validate inputs
3. The browser constructs an HTTP request
4. Credentials are placed into the POST body
5. The request is sent to the server

The server then:
- Processes the credentials
- Verifies them
- Grants or denies access

---

# Hydra http-post-form Module

Hydra provides:
```text
http-post-form
```

specifically for attacking login forms.

This module automates:
- HTTP POST requests
- Credential insertion
- Response analysis

allowing attackers to test:
```text
Large numbers of username/password combinations
```

very efficiently.

---

# General Hydra Syntax

```bash
hydra [options] target http-post-form "path:params:condition"
```

---

# Understanding the Hydra Structure

The command contains:
- Target
- Login path
- Form parameters
- Success/failure condition

Hydra dynamically replaces:
```text
^USER^
^PASS^
```

with credentials from wordlists.

---

# Failure Conditions vs Success Conditions

Hydra determines whether authentication succeeded by analyzing:
- Response messages
- HTTP status codes
- Redirects
- Page content

---

# Failure Condition

Most websites display an error message when login fails.

Example:

```text
Invalid credentials
```

Hydra can use this as a:
```text
Failure condition
```

---

# Example Failure Condition

```bash
hydra ... http-post-form "/login:user=^USER^&pass=^PASS^:F=Invalid credentials"
```

Hydra interprets:
```text
F=Invalid credentials
```

as:
```text
If this message appears, login failed
```

Hydra then continues testing new credentials.

---

# Success Condition

Sometimes websites:
- Redirect users after login
- Display dashboard pages
- Show welcome messages

Instead of looking for failure messages, Hydra can look for:
```text
Success indicators
```

---

# Example Using HTTP Redirect

```bash
hydra ... http-post-form "/login:user=^USER^&pass=^PASS^:S=302"
```

This means:
```text
HTTP 302 redirect indicates successful login
```

---

# Example Using Dashboard Keyword

```bash
hydra ... http-post-form "/login:user=^USER^&pass=^PASS^:S=Dashboard"
```

Hydra treats:
```text
Dashboard
```

as proof of successful authentication.

---

# Gathering Login Form Information

Before using Hydra, attackers must understand:
- Form structure
- Parameters
- Request method
- Response behavior

This process is called:
```text
Reconnaissance
```

---

# Manual Inspection

Using browser developer tools:
- Right-click page
- Click "Inspect"

allows us to view the HTML source.

Example:

```html
<form method="POST">

<input type="text" name="username">

<input type="password" name="password">

</form>
```

From this we identify:
- Request method = POST
- Username parameter = username
- Password parameter = password

These values are critical for Hydra.

---

# Browser Developer Tools

Open:
```text
F12 → Network tab
```

Submit a fake login request.

The browser will display:
- HTTP request
- Headers
- POST data
- Response

This reveals:
- Exact target path
- Parameter names
- Response messages

which are needed to construct a working Hydra command.

---

# Proxy Interception

For more advanced analysis, tools like:
- Burp Suite
- OWASP ZAP

can intercept traffic between:
- Browser
- Web server

This allows attackers to inspect:
- Hidden parameters
- Tokens
- Headers
- CSRF values
- Cookies

# Constructing the params String

The:
```text
params string
```

is one of the most important parts of Hydra.

General structure:

```text
path:parameters:condition
```

---

# Example params String

```bash
/:username=^USER^&password=^PASS^:F=Invalid credentials
```

---

# Breaking It Down

## /

This is:
```text
The login path
```

meaning the login form submits to:
```text
/
```

---

## username=^USER^

Hydra replaces:
```text
^USER^
```

with usernames from the username list.

---

## password=^PASS^

Hydra replaces:
```text
^PASS^
```

with passwords from the password list.

---

## F=Invalid credentials

Hydra interprets:
```text
Invalid credentials
```

as:
```text
Failed authentication
```

---

# Final Hydra Command

```bash
hydra -L top-usernames-shortlist.txt \
-P 2023-200_most_used_passwords.txt \
-f IP -s 5000 \
http-post-form \
"/:username=^USER^&password=^PASS^:F=Invalid credentials"
```

---

# Understanding the Command

## -L

```bash
-L top-usernames-shortlist.txt
```

Loads:
```text
Username wordlist
```

---

## -P

```bash
-P 2023-200_most_used_passwords.txt
```

Loads:
```text
Password wordlist
```

---

## -f

```bash
-f
```

Stops Hydra after:
```text
First valid credential pair
```

is discovered.

---

## -s 5000

Specifies:
```text
Custom target port
```

---

## http-post-form

Tells Hydra:
```text
Attack a web login form using POST requests
```

- Valid username

# Why Login Forms Are Common Targets

Web login forms are frequently attacked because:
- They are publicly accessible
- Users often reuse passwords
- Weak credentials are common
- Default credentials may exist

Attackers commonly use:
- Credential stuffing
- Password spraying
- Brute force attacks

against login portals.

---

# Important Security Defenses

Modern applications defend against brute-force attacks using:
- Rate limiting
- CAPTCHA
- MFA
- Account lockout
- WAF protection
- IP blocking
- Request throttling

Without these protections:
```text
Hydra attacks can become extremely effective
```

---

# Key Learning Points

- Login forms send credentials using HTTP requests
- Hydra's `http-post-form` module automates brute-force attacks against web login forms
- Understanding parameter names and request structure is essential
- Browser Developer Tools and Burp Suite help analyze login requests
- Hydra relies on success or failure conditions to identify valid logins
- Properly crafting the params string is critical for successful attacks
