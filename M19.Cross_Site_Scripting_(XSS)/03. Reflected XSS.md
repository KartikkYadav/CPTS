# Reflected XSS (Non-Persistent XSS) - CPTS Notes

## Introduction

Reflected XSS is one of the two major types of **Non-Persistent Cross-Site Scripting (XSS)** vulnerabilities. Unlike Stored XSS, where the payload is permanently saved in the database, Reflected XSS only exists during a single request and response cycle.

The attacker's payload is sent to the web server, processed by the application, and immediately reflected back in the HTTP response without being stored anywhere. Because the payload is temporary, it disappears once the page is refreshed or revisited.

This means that Reflected XSS usually affects only the specific user who clicks a malicious link or submits attacker-controlled input.

---

# What is Reflected XSS?

Reflected XSS occurs when user-supplied input reaches the backend server and is returned to the browser without proper sanitization or encoding.

The application effectively "reflects" the input back to the user.

Typical flow:

```text
Attacker Input
      ↓
Server Receives Input
      ↓
Application Processes Input
      ↓
Input Reflected In Response
      ↓
Browser Executes JavaScript
```

Since the payload is not stored in a database, it exists only for that single response.

---

# Difference Between Stored and Reflected XSS

### Stored XSS

```text
Attacker Input
      ↓
Database Storage
      ↓
Victim Visits Page
      ↓
Payload Executes
```

The payload remains stored and affects everyone visiting the page.

---

### Reflected XSS

```text
Attacker Input
      ↓
Server Processes Request
      ↓
Payload Returned Immediately
      ↓
Browser Executes Payload
```

The payload is never stored and disappears after the request is completed.

---

# Example Scenario

Consider a To-Do application.

When a user submits:

```text
test
```

The application responds with:

```text
Task 'test' could not be added.
```

Notice that the application includes the user's input inside the response.

This immediately tells us that our input is being reflected back to the page.

Whenever user-controlled input appears in server responses, XSS testing should begin.

---

# Testing For Reflected XSS

A common XSS testing payload is:

```html
<script>alert(window.origin)</script>
```

This payload is useful because:

* It is easy to identify visually.
* It confirms JavaScript execution.
* It displays the domain where the code executes.

---

# Submitting The Payload

Instead of entering:

```text
test
```

submit:

```html
<script>alert(window.origin)</script>
```

If the application is vulnerable:

```text
Popup Appears
```

showing:

```text
http://target-site.com
```

This confirms successful JavaScript execution.

---

# Understanding The Result

After submission, the page may display:

```text
Task '' could not be added.
```

At first glance it may appear that the payload disappeared.

However, this happens because:

```html
<script>alert(window.origin)</script>
```

is interpreted as an HTML element rather than normal text.

The browser executes the script instead of displaying it.

---

# Verifying Through Page Source

To confirm the payload is reflected:

```text
CTRL + U
```

or

```text
Right Click → View Page Source
```

Example:

```html
<div style="padding-left:25px">
Task '<script>alert(window.origin)</script>' could not be added.
</div>
```

The payload is clearly present inside the response.

The browser simply interpreted it as executable JavaScript.

---

# Why Is It Called Non-Persistent?

After refreshing the page:

```text
Refresh Page
     ↓
Message Disappears
     ↓
Payload Disappears
     ↓
No Alert
```

The payload is gone because:

* It was never stored.
* It only existed in that response.
* A new request generates a clean page.

This is the defining characteristic of Reflected XSS.

---

# Why Is Reflected XSS Dangerous?

A common question is:

> If the payload disappears after refresh, how can attackers exploit it?

The answer is:

```text
Malicious URLs
```

The attacker places the payload inside a request parameter and sends the URL to victims.

When the victim clicks the link, the application reflects the payload and the browser executes it.

---

# Finding The Request Method

To understand how the payload is transmitted:

Open Developer Tools:

```text
CTRL + SHIFT + I
```

Navigate to:

```text
Network Tab
```

Submit the payload again.

Observe the request.

Example:

```http
GET /?task=<script>alert(window.origin)</script>
```

If the request method is:

```http
GET
```

the payload becomes part of the URL.

---

# Why GET Requests Matter

GET requests place user data directly inside the URL.

Example:

```text
http://target.com/?task=<script>alert(window.origin)</script>
```

An attacker can simply send this URL to a victim through:

* Email
* Chat messages
* Social media
* Phishing pages
* QR codes

When the victim opens the URL:

```text
Victim Clicks Link
         ↓
Request Sent To Server
         ↓
Payload Reflected
         ↓
Browser Executes JavaScript
```

The attack succeeds.

---

# Typical Reflected XSS Attack Flow

```text
1. Find Reflected Input
2. Inject JavaScript Payload
3. Confirm Execution
4. Identify Request Method
5. Generate Malicious URL
6. Send URL To Victim
7. Victim Opens URL
8. Browser Executes Payload
```

Unlike Stored XSS, every victim must interact with the crafted link.

---

# Common Locations For Reflected XSS

Reflected XSS frequently appears in:

### Search Pages

```text
Search Results For: test
```

---

### Error Messages

```text
User 'test' Not Found
```

---

### Login Errors

```text
Invalid Username: test
```

---

### Contact Forms

```text
Message From: test
```

---

### Confirmation Messages

```text
Task 'test' Could Not Be Added
```

Any location where user input is displayed back to the user should be tested.

---

# Common Impacts

Even though Reflected XSS is temporary, it can still have severe consequences.

### Session Theft

```javascript
document.cookie
```

Steal session cookies.

---

### Account Takeover

Use stolen sessions to access victim accounts.

---

### Credential Harvesting

Inject fake login forms.

---

### Sensitive Data Theft

Read data visible to the victim.

---

### Unauthorized Actions

Perform actions using the victim's authenticated session.

---

### Browser Redirection

Redirect victims to phishing websites.

---

# Reflected XSS vs Stored XSS

| Feature                     | Stored XSS     | Reflected XSS |
| --------------------------- | -------------- | ------------- |
| Stored In Database          | Yes            | No            |
| Survives Refresh            | Yes            | No            |
| Affects Every Visitor       | Yes            | No            |
| Requires Victim Interaction | Usually No     | Usually Yes   |
| Delivery Method             | Stored Content | Crafted URL   |
| Severity                    | Higher         | Medium        |

---

# Detection Methodology

When testing applications:

```text
1. Find User Input
2. Submit Test String
3. Verify Reflection
4. Submit XSS Payload
5. Observe Execution
6. Refresh Page
7. Check Persistence
8. Analyze Request Type
```

If the payload executes but disappears after refresh:

```text
Reflected XSS Confirmed
```

# Quick Summary

```text
Attacker Creates Payload
            ↓
Payload Added To URL
            ↓
Victim Opens Link
            ↓
Server Reflects Input
            ↓
Browser Executes JavaScript
            ↓
Reflected XSS
```

Reflected XSS is a Non-Persistent XSS vulnerability where user input is reflected back from the server without proper sanitization. Although the payload is not stored, it can still lead to session theft, account compromise, phishing attacks, and unauthorized actions when victims are tricked into visiting malicious URLs.
