# DOM-Based XSS - CPTS Notes

## Introduction

DOM-Based XSS is the third major type of Cross-Site Scripting vulnerability and is classified as a **Non-Persistent XSS** vulnerability.

Unlike Stored XSS and Reflected XSS, DOM XSS does not require the payload to be processed by the backend server. The entire attack occurs within the victim's browser through client-side JavaScript.

This makes DOM XSS fundamentally different because the malicious payload never reaches the web server. Instead, JavaScript running in the browser reads attacker-controlled input and dynamically modifies the page content, causing malicious code to execute.

---

# What Makes DOM XSS Different?

Let's compare all three XSS types:

| XSS Type      | Backend Involved | Stored | Processed By     |
| ------------- | ---------------- | ------ | ---------------- |
| Stored XSS    | Yes              | Yes    | Server + Browser |
| Reflected XSS | Yes              | No     | Server + Browser |
| DOM XSS       | No               | No     | Browser Only     |

The most important thing to remember:

```text
Stored XSS
→ Payload stored in database

Reflected XSS
→ Payload reflected by server

DOM XSS
→ Payload processed entirely in browser
```

---

# Understanding DOM XSS

A DOM XSS vulnerability occurs when client-side JavaScript takes user-controlled input and writes it into the page without proper sanitization.

Typical flow:

```text
Attacker Input
      ↓
JavaScript Reads Input
      ↓
JavaScript Modifies DOM
      ↓
Browser Executes Payload
```

The server never sees the malicious payload.

---

# Example Scenario

Consider a To-Do List application.

Adding a task:

```text
test
```

Results in:

```text
Next Task: test
```

At first glance everything appears normal.

However, when we open the browser's Developer Tools and inspect the Network tab, something unusual happens:

```text
No HTTP Requests Are Sent
```

This is a strong indicator that the application is processing the data entirely on the client side.

---

# Identifying DOM-Based Input

The URL might look like:

```text
http://target.com/#task=test
```

Notice the:

```text
#
```

character.

Everything after the hashtag is called a:

```text
URL Fragment
```

or

```text
Anchor
```

Data inside the fragment:

```text
#task=test
```

is never sent to the server.

Instead:

```text
Browser Reads Fragment
       ↓
JavaScript Processes Fragment
       ↓
DOM Updates Page
```

This is why DOM XSS vulnerabilities often use:

* URL fragments (#)
* Client-side parameters
* JavaScript variables
* Browser storage

instead of normal HTTP requests.

---

# Why No Network Requests Appear

With Reflected XSS:

```text
Input
 ↓
HTTP Request
 ↓
Server
 ↓
Response
```

With DOM XSS:

```text
Input
 ↓
Browser
 ↓
JavaScript
 ↓
DOM Update
```

No communication with the server occurs.

This is one of the easiest ways to recognize a DOM XSS vulnerability.

---

# Why Input Doesn't Appear In Page Source

If we press:

```text
CTRL + U
```

we may not find our input.

Reason:

```text
Page Source
=
Original HTML
```

DOM modifications happen after the page loads.

The browser receives:

```html
<div id="todo"></div>
```

Then JavaScript modifies it later.

Because of this:

```text
View Source
≠
Current Rendered Page
```

---

# Viewing The Modified DOM

To see dynamically generated content:

```text
CTRL + SHIFT + C
```

or use:

```text
Web Inspector
```

This shows the live DOM after JavaScript modifications.

---

# Source and Sink Concept

Understanding DOM XSS requires understanding:

```text
Source
```

and

```text
Sink
```

---

# Source

The Source is where attacker-controlled data originates.

Examples:

```text
URL Parameters
Hash Fragments
Input Fields
Cookies
Local Storage
Session Storage
```

Example:

```javascript
var pos = document.URL.indexOf("task=");
var task = document.URL.substring(pos + 5);
```

Here:

```text
document.URL
```

is the Source because it contains user-controlled input.

---

# Sink

The Sink is where the input is written into the page.

Example:

```javascript
document.getElementById("todo").innerHTML =
"<b>Next Task:</b> " + decodeURIComponent(task);
```

Here:

```text
innerHTML
```

is the Sink.

The user input flows from:

```text
Source
 ↓
JavaScript
 ↓
Sink
```

If the Sink writes data without sanitization, DOM XSS becomes possible.

---

# Common Dangerous Sink Functions

## Native JavaScript Sinks

```javascript
document.write()
```

```javascript
element.innerHTML
```

```javascript
element.outerHTML
```

These functions directly insert HTML into the page.

---

## Common jQuery Sinks

```javascript
add()
```

```javascript
after()
```

```javascript
append()
```

These functions can also introduce unsanitized user input into the DOM.

---

# Identifying The Vulnerability

Consider:

```javascript
var task = document.URL.substring(pos + 5);
```

Input comes from:

```text
#task=
```

parameter.

Then:

```javascript
document.getElementById("todo").innerHTML =
"<b>Next Task:</b> " + decodeURIComponent(task);
```

writes the value directly into the page.

No sanitization occurs.

This creates a DOM XSS vulnerability.

---

# Why Normal Script Tags Fail

Most beginners try:

```html
<script>alert(window.origin)</script>
```

However:

```text
innerHTML
```

contains built-in protections that prevent many `<script>` tags from executing.

As a result:

```html
<script>
```

often fails in DOM XSS scenarios.

---

# Alternative Payload Using HTML Events

Instead of using script tags, attackers often use HTML event handlers.

Example:

```html
<img src="" onerror=alert(window.origin)>
```

---

# How This Payload Works

The browser attempts to load:

```html
<img src="">
```

Because the image does not exist:

```text
Image Loading Fails
```

This triggers:

```html
onerror=
```

which executes JavaScript.

Result:

```javascript
alert(window.origin)
```

executes successfully.

---

# Example Attack URL

```text
http://target.com/#task=<img src='' onerror=alert(window.origin)>
```

When the victim visits the URL:

```text
Browser Loads Page
       ↓
JavaScript Reads URL Fragment
       ↓
innerHTML Writes Payload
       ↓
Image Fails To Load
       ↓
onerror Executes
       ↓
Alert Appears
```

No server interaction occurs at any point.

---

# Delivering The Attack

Since the payload is contained inside the URL:

```text
Attacker Creates URL
        ↓
Sends URL To Victim
        ↓
Victim Opens URL
        ↓
Payload Executes
```

Delivery methods include:

* Email
* Chat applications
* Social media
* QR codes
* Phishing pages


# Stored vs Reflected vs DOM XSS

| Feature                 | Stored     | Reflected   | DOM         |
| ----------------------- | ---------- | ----------- | ----------- |
| Stored In Database      | Yes        | No          | No          |
| Backend Processing      | Yes        | Yes         | No          |
| Browser Processing      | Yes        | Yes         | Yes         |
| Persistent              | Yes        | No          | No          |
| Requires Malicious Link | Usually No | Usually Yes | Usually Yes |

# Quick Summary

```text
Attacker Controls URL Fragment
            ↓
JavaScript Reads Input
            ↓
Data Flows From Source
            ↓
Dangerous Sink Writes Input
            ↓
Browser Executes Payload
            ↓
DOM-Based XSS
```

DOM-Based XSS is a client-side XSS vulnerability where malicious input never reaches the backend server. Instead, JavaScript running in the browser reads attacker-controlled data and inserts it into the DOM through unsafe Sink functions, resulting in JavaScript execution.
